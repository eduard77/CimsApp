# CIMS v1.1 backlog

Per PAFM-SD commitment C-03: "I will keep a v1.1 backlog so
deferred work is visible and cannot be 'lost'." Items here are
out of scope for v1.0 but formally acknowledged as future work.

This is not a to-do list. Items move into a future sprint only
after v1.0 ships and a sprint kickoff has considered them against
the Chapter 17 change control process.

## How to use this file

- **Add an item** when a feature is consciously deferred — either
  out-of-scope per PAFM Ch 3, or identified as hardening work that
  exceeds the current sprint's budget.
- **Every item** has: source (what discovered it), category,
  short description, and why it is deferred rather than closed now.
- **Removing an item** happens when it lands in a sprint. Link the
  removal to the implementing commit in the commit message.
- Items are not prioritised here; prioritisation happens at v1.1
  planning time using the current PAFM and updated success criteria.

### ~~B-002 — Rate limiting and abuse protection on anonymous endpoints~~ — CLOSED 2026-04-28

**Source:** Security review SR-S0-06.
**Category:** Platform hardening.
**Description.** `POST /auth/register`, `POST /auth/login`,
`POST /auth/refresh`, and `POST /organisations` have no rate limit,
CAPTCHA, or email verification. Add ASP.NET Core rate-limiting
middleware, progressive back-off on login failures, CAPTCHA or
email-verification on organisation creation.

**Status (2026-04-28, partial close).** Rate-limiting half landed
on branch `feature/b-002-rate-limiting` via ASP.NET Core's built-in
`Microsoft.AspNetCore.RateLimiting`. Two named policies:
- `anon-login`: 5 req/min per IP — applied to `/auth/login` only
  (credential-testing target).
- `anon-default`: 10 req/min per IP — applied to `/auth/register`,
  `/auth/refresh`, and anonymous `POST /organisations`.

Fixed-window partitioning by `RemoteIpAddress`. 429 returned on
limit hit. Manual smoke verified: 5 sequential `POST /auth/login`
return application errors (expected — no users seeded), the 6th
returns 429 from the limiter before the endpoint executes.

**Closed 2026-04-28.** Progressive back-off shipped on branch
`feature/b-002-progressive-backoff-and-b-003-obsolete`:

- `CimsApp.Services.Auth.LoginAttemptTracker` (singleton, backed
  by `IMemoryCache`). Per-IP failed-attempt counter with a
  15-minute sliding window. Lockout threshold 5.
- `AuthService.LoginAsync` short-circuits with `AppException(429,
  "LOGIN_LOCKOUT")` BEFORE any DB lookup when the caller's IP is
  locked out — credential-stuffing campaigns burn through their
  rate-limit budget without learning anything new.
- `try/catch (AppException ex when ex.StatusCode == 401)` wraps
  the auth flow: only 401-class credential failures count toward
  lockout (validation errors do not — operator typos shouldn't
  tighten the screw). Successful login resets the counter.

11 new tests across two files (185 / 185 suite green):
`LoginAttemptTrackerTests` covers the tracker rules in isolation;
`LoginProgressiveBackOffTests` covers the AuthService integration
with both stub-and-real-tracker paths, including a five-failure
end-to-end that confirms even valid credentials are rejected
once locked.

CAPTCHA on organisation creation and email verification remain
out of scope for v1.0 — neither is required by the internal
pilot. Each warrants its own backlog item if the pre-customer
posture demands them; promotion happens at v1.1 planning.

### ~~B-003 — Constant-time register path~~ — OBSOLETE 2026-04-28

**Source:** Security review SR-S0-07.
**Category:** Auth hardening.
**Description.** `AuthService.RegisterAsync` first checks email
existence, then organisation existence. Differing latencies leak a
small email-enumeration oracle.

**Resolution.** The oracle described here was eliminated as a
side effect of the **T-S0-11 invitation-token rewrite** (closes
SR-S0-01, ADR-0011). The post-T-S0-11 RegisterAsync flow is:

  1. ValidateAsync(invitationToken, email) — single
     `Invitations.FirstOrDefaultAsync(i => i.TokenHash == hash)`
     followed by in-memory checks (consumed, expired, email
     bind). One DB hit regardless of which check fails.
  2. If invitation valid: check email duplicate; find org;
     persist user.

A caller without a valid invitation token NEVER reaches the
email duplicate check or the organisation lookup — there is no
timing differential between "valid email" and "invalid email"
because the duplicate check is gated behind a successful
invitation validation. The email-enumeration vector this item
described is closed by the gate, not by constant-time
arithmetic.

No code change required to retire this item. Filed obsolete
rather than closed because the work envisaged was never the
work that landed.

### ~~B-017 — Per-CBS-line schedule + progress signal~~ — CLOSED 2026-04-27

**Source:** S1 retrospective (`docs/retrospectives/s1.md`).
Consolidates three service-integration deferrals taken
independently across S1.

**Closed 2026-04-27.** All three downstream service-integration
wire-ups landed on branch `feature/b-017-line-schedule-progress`
immediately after the MudBlazor 9 upgrade. PR #4.

Primitive shipped:
- `ScheduledStart`, `ScheduledEnd`, `PercentComplete decimal(5,4)
  ∈ [0, 1]` columns on `CostBreakdownItem`.
- `SetLineScheduleAsync` and `SetLineProgressAsync` service
  methods plus two PM+ endpoints
  (`PUT /cbs/{itemId}/schedule`, `PUT /cbs/{itemId}/progress`).
- Audit-twin actions `cbs.line_schedule_set` and
  `cbs.line_progress_set`.

Three downstream wire-ups shipped:
- **T-S1-07 EVM service integration.**
  `CostService.GetEvmSnapshotAsync(projectId, dataDate)` walks
  every CBS line summing BAC = Σ Budget, EV = Σ Budget ×
  PercentComplete, PV = Σ Budget × scheduleFraction(dataDate,
  ScheduledStart, ScheduledEnd). AC = Σ ActualCost. Feeds
  `Evm.Calculate` from T-S1-07 unchanged. New endpoint
  `GET /evm?dataDate=...`.
- **T-S1-09 payment-cert valuation auto-derive.**
  `PaymentCertificateDto` gains a computed
  `DerivedValuationFromProgress = Σ Budget × PercentComplete`
  field. Stored `CumulativeValuation` remains the assessor-stated
  source of truth per ADR-0013; the derived field is the
  progress-based guide alongside it.
- **T-S1-11 per-CBS-line cashflow breakdown.**
  `CostService.GetCashflowByLineAsync` returns one
  `CashflowLineSeries` per CBS line, each with one
  `CashflowLinePeriodPoint` per CostPeriod. BaselinePlanned per
  point = Budget × overlap-fraction(period range, schedule
  range). Actual per point = Σ ActualCost where line + period
  match. New endpoint `GET /cashflow/by-line`.

146 / 146 suite green at sprint-close commit on the branch.
**Category:** Cost-domain data primitive (v1.1).
**Description.** v1.0 has no per-CBS-line scheduled-spend curve
and no per-CBS-line progress (percent-complete) signal. The
absence blocks three otherwise-shippable service integrations:

- **T-S1-07 EVM service integration.** `CimsApp/Core/Evm.cs`
  ships the pure math (PV/EV/AC → CPI/SPI/EAC/TCPI). A
  service that reads project data and produces an
  `EvmSnapshot` needs PV (planned value at data date — needs
  the schedule baseline) and EV (earned value — needs
  progress).
- **T-S1-09 payment-certificate valuation auto-derivation.**
  `PaymentCertificate.CumulativeValuation` is currently
  manually entered by the assessor. With per-line progress,
  it could be auto-derived as `Σ (CBS line Budget × percent
  complete)` per ADR-0013 NEC4 PWDD.
- **T-S1-11 per-CBS-line cashflow breakdown.**
  `CostService.GetCashflowAsync` returns project-level
  totals only. Per-line cashflow needs each line's planned
  spend distributed across periods, which is the same
  schedule-baseline data the EVM PV needs.

A minimum useful schema would add:

- `CostBreakdownItem.ScheduledStart` / `ScheduledEnd` (date
  range — when work on this line is planned).
- `CostBreakdownItem.PercentComplete` (0..100, or stored as
  decimal 0..1).
- A method (linear by default) to distribute a line's Budget
  across its scheduled date range, intersected with project
  CostPeriods, to produce the planned-cashflow contribution
  per (line, period).

All three integrations then become read-only computations on
top of this primitive. No service rewrites are needed; just
new methods that consume the new fields.

**Why deferred.** Adding the primitive would have inflated the
S1 scope by another ~8-12h (entity columns, migration,
service updates across three sites, behavioural tests) on top
of the CR-003-trimmed 96h budget that already sits 24h over
formal capacity. Each downstream integration was deferred
locally with a forward-pointing comment; this item rolls them
up so the v1.1 planner sees the cross-cutting work as one
decision rather than three.

### B-018 — `LoginAttemptTracker` is single-instance only

**Source:** B-002 close (2026-04-28). Self-flagged at the
landing commit.
**Category:** Platform hardening / horizontal-scale readiness.
**Description.** The progressive back-off counter
(`CimsApp.Services.Auth.LoginAttemptTracker`) is backed by
`IMemoryCache` registered as a process-singleton in `Program.cs`.
That works for the v1.0 deployment shape (single web process)
but does not survive horizontal scale-out: each replica would
have its own counter, so an attacker could spread credential
testing across replicas and avoid the lockout entirely.

**Why deferred.** v1.0 deploys as a single instance; the
limitation has zero exposure today. When (and if) the
deployment scales horizontally, the natural fixes are (a)
move the counter into a shared distributed cache (Redis,
SQL Server in-memory OLTP) or (b) move the counter onto the
DB as an `Authn` table with row-level locking. (a) is faster
and cheaper; (b) gives free durability across restarts.

Pre-customer hardening: revisit before any deployment
topology that includes more than one `CimsApp` web process.

### B-024 — Optimistic concurrency control on mutable entities

**Source:** Post-S1 secondary-mutation audit
(`docs/security/post-s1-secondary-mutation-audit-2026-04-29.md`,
finding SM-04).
**Category:** Data integrity / concurrent-edit safety.
**Description.** No entity in `CimsApp/Models/Entities.cs`
carries a rowversion / `IsConcurrencyToken` column. Two
concurrent edits of the same project / payment cert / CBS
line are silent last-write-wins — the loss is forensically
reconstructable via `AuditInterceptor` but the user
experience is silent data loss.

**Scope of work.**

- Add `[Timestamp] public byte[] RowVersion { get; set; }`
  (or EF-configured equivalent) to every entity with a UI
  edit path: Project, ProjectMember, CdeContainer, Document,
  DocumentRevision, Rfi, ActionItem, ProjectTemplate,
  CostBreakdownItem, Commitment, CostPeriod, ActualCost,
  Variation, PaymentCertificate.
- One migration adding the column to all of them.
- `ErrorHandlingMiddleware` maps
  `DbUpdateConcurrencyException` → 409 with a structured
  body describing the conflict.
- API responses surface the rowversion as an `ETag` header;
  PUT/PATCH callers send `If-Match`. Server compares; mismatch
  short-circuits to 409 before the DB write.
- UI-side conflict-resolution flow — at minimum a "this
  record changed since you opened it; reload?" dialog;
  ideally a side-by-side merge for high-value entities like
  PaymentCertificate.

**Estimate.** ~12-16h in a sprint.

**Why deferred.** v1.0 internal pilot has low concurrency;
the loss probability is tiny. Becomes load-bearing for v1.1
once concurrent users become a normal operational shape.
The audit-twin pattern's per-row Update capture means a
silently-overwritten edit IS reconstructable — so the
exposure is "user does work that vanishes" rather than "user
loses unrecoverable state". Acceptable trade for v1.0.

the comment "UI-layer noise", suggesting the intent was a
high-volume UI notification stream that shouldn't pollute the
audit log).

**Status today.** Zero producers, zero consumers. The table is
defined, the DbSet exists, the tenant query filter is wired,
but no service writes to it and no controller serves it. It
has been dormant for the entire project lifetime.

**Scope of work to make it real.**

- A `NotificationService` with `CreateAsync(userId, type,
  title, body, link)` and `MarkReadAsync(notificationId,
  userId)` methods.
- Hook the create from the obvious producers — assigned an
  RFI / Action, mentioned in a comment, document transitions
  to a state the user should know about, payment cert issued
  to the user's org. Each producer needs explicit thinking
  about *should this notify*.
- A `NotificationsController` with `GET` (list, scoped to
  `User.Id == CurrentUserId`), `POST {id}/read`.
- Real-time delivery (SignalR? polling?) is a separate
  decision — v1.1 might ship polling-only.

**Why deferred.** v1.0 internal pilot does not need
notifications — a single internal user works through the UI
synchronously. The feature has real product surface area
(per-event notification policy, real-time delivery, read /
unread / dismissed lifecycle, mobile push later) that needs
its own sprint, not an opportunistic add.

**Recommendation when picked up.** Do NOT just delete the
entity to "clean up". The schema scaffold is harmless (empty
table) and re-adding the migration when the feature lands
creates churn. Leave dormant; pick it up as a real sprint.

### B-026 — `ProjectAppointment` / B2B contractor membership

**Source:** Post-S1 hardening pass (2026-04-29) — surfaced
alongside B-025.
**Category:** Multi-org collaboration (post-v1.0).
**Description.** A `ProjectAppointment` entity links a
non-AppointingParty Organisation to a Project — the original
intent (per the post-S1 secondary-mutation audit, SM-03's
"future expansion") was that v1.1 would let an
AppointingParty *appoint* a contractor org onto their
project, and users in the appointed org would gain project
access without being members of the AppointingParty
organisation.

**Status today.** Zero producers. `ProjectsService.GetByIdAsync`
loads `p.Appointments` via Include but the collection is
always empty. The entity, the FK relationships, the tenant
query filter, and the `Organisation.Appointments` /
`Project.Appointments` navigation properties all exist
unused.

**Scope of work to make it real.**

- A "create appointment" service method + controller endpoint:
  `POST /api/v1/projects/{projectId}/appointments` with body
  `{ organisationId, role, startDate?, endDate? }`. OrgAdmin
  of the AppointingParty (or SuperAdmin) only.
- Widen the B-023 / `AddMemberAsync` org-match check from
  "member's org == AppointingParty" to "member's org ==
  AppointingParty OR has an active ProjectAppointment on
  this project". The post-S1 secondary-mutation audit
  flagged this future widening explicitly.
- Tenant query filter widening: a user in an appointed org
  needs to *see* the project (currently the filter is
  `Project.AppointingPartyId == _tenant.OrganisationId`,
  which would hide it). Two-tenant visibility is the real
  architectural change — needs its own ADR.

**Why deferred.** v1.0 model is single-org-per-project per
ADR-0012 ("Project tenancy: caller's-org default with
SuperAdmin bypass"). Multi-org membership crosses an
ADR-level architectural boundary; an ADR is required to
update ADR-0012 (or supersede with a new ADR) before
implementation lands.

**Recommendation when picked up.** Same as B-025: leave the
schema dormant. The entity is part of the eventual
architecture; removing it would just create churn when the
feature lands.

### B-027 — SQL Server smoke test in CI / pre-merge

**Source:** Post-S1 hardening session 2026-04-29 — concrete
incident, not theoretical. The PR #41 bug
(`AuditLog.UserId` non-nullable + Guid.Empty sentinel for
anonymous flows = FK violation on real SQL Server) was 100%
latent under unit-test-only validation because the EF
in-memory provider ignores FK constraints. Was broken in
production for ~a week (since T-S0-11 / PR #11) before a
human-driven smoke test caught it.

**Category:** CI / quality gate.

**Description.** Add a CI job that boots the app against a
real SQL Server instance (LocalDB on Windows runners,
mssql/server in Docker on Linux runners), applies all
migrations, and exercises a small set of smoke endpoints:

- `POST /api/v1/organisations` (anonymous, returns 201 with
  bootstrap invitation) — covers the FK-permissive class of
  bug PR #41 closed.
- `POST /api/v1/auth/register` with the bootstrap token —
  covers the RegisterAsync transaction wrap (PR #29).
- `POST /api/v1/auth/login` with the new credentials — full
  auth path through SQL Server.
- `GET /api/v1/projects` — authenticated read with the
  tenant filter active.
- `POST /api/v1/projects` — authenticated write that goes
  through audit-twin atomicity (PR #33).

Each smoke test asserts the HTTP status and basic response
shape. Failures must block merge.

**Why deferred.** Setting up a CI job with a SQL Server
service is non-trivial — plus deciding the CI runner shape
(GitHub Actions Windows for LocalDB? Linux + mssql? both?).
Worth doing before any external customer deploy because the
in-memory test gap is not theoretical: it just bit us once.

**Until then:** treat the smoke-test-against-SQL-Server step
as a pre-deploy human checkpoint per the post-S1 handoff
doc's forward path #3.

### B-022 — Cache the `OnTokenValidated` user lookup

**Source:** Post-S1 auth-mutation audit
(`docs/security/post-s1-auth-mutation-audit-2026-04-28.md`,
finding AM-05).
**Category:** Performance / horizontal-scale readiness.
**Description.** `JwtBearerEvents.OnTokenValidated` does a
per-request `db.Users.FirstOrDefaultAsync` lookup to run
`TokenRevocation.IsRevoked`. Acceptable for the v1.0
single-instance deployment (PK lookup, fully indexed). Becomes
a hot path under horizontal scale (B-018 territory). Natural
fixes:

- Short-TTL in-memory cache (e.g. 30 s) per user, invalidated
  explicitly on `RevokeUserTokensAsync`.
- Per-user version stamp denormalised onto the JWT — the hook
  compares the stamp against the current User row, only doing
  the DB lookup when the stamps disagree.
- Distributed cache (paired with B-018's distributed
  `LoginAttemptTracker`).

**Why deferred.** Premature for v1.0 single-instance traffic.
Revisit alongside B-018 when scale-out is planned.

### B-028 — Schedule-side Monte Carlo simulation

**Source:** PAFM-SD `.docx` Appendix F.3; CR-004 deferred from S2.
**Category:** Risk simulation / S2 fast-follow OR v1.1.
**Description.** PAFM-SD Appendix F.3 names "Monte Carlo simulation

**Why deferred.** Capacity (CR-004) plus a real upstream dependency

### B-029 — Opportunity register (mirror of Risk)

**Source:** PAFM-SD `.docx` Appendix F.3; CR-004 deferred from S2.
**Category:** Risk module completion.
**Description.** PAFM-SD F.3 closes with "Opportunity register with
the same structure" — a structural copy of the Risk register that
tracks positive risks (events that, if they occur, improve project
outcomes). Same RBS taxonomy, same 5×5 P×I matrix, same response
strategies (exploit / share / enhance / accept rather than the
threat-side avoid / transfer / mitigate / accept).
**Why deferred.** Mostly mechanical mirroring of Risk; doubles the
test surface for marginal v1.0 pilot value. Most UK construction
risk processes track threats (downside) far more rigorously than
opportunities (upside). Re-evaluate at first external customer who
manages positive risks formally.

### B-030 — Cross-module contingency drawdown link

**Source:** PAFM-SD `.docx` Appendix F.3 (drawdown tracking bullet);
CR-004 deferred from S2.
**Category:** Risk → Cost & Commercial integration.
**Description.** v1.0 (T-S2-09) tracks contingency drawdown amounts
per Risk via manual entry. The richer model links each drawdown
event to the specific Commitment / ActualCost / Variation that
consumed the contingency, giving a forensic chain from "risk
materialised" to "money spent". Crosses the Risk module / Cost &
Commercial module boundary; needs a service-layer integration
point and a UI that lets the user pick the consumer at drawdown
time.
**Why deferred.** Capacity plus a clean separation-of-concerns
benefit: v1.0 ships drawdown numbers honestly without coupling the
Risk module to S1's mutation surface. Re-evaluate when actual
construction projects start using the system and demand the
forensic chain (likely first external customer).
