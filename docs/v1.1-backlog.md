# CIMS v1.1 backlog

Per PAFM-SD commitment C-03: "I will keep a v1.1 backlog so
deferred work is visible and cannot be 'lost'." Items here are
out of scope for v1.0 but formally acknowledged as future work.

This is not a to-do list. Items move into a future sprint only
after v1.0 ships and a sprint kickoff has considered them against
the Chapter 17 change control process.

## How to use this file

- **Add an item** when a feature is consciously deferred — either
  out-of-scope per PAFM Ch 3, or identified as hardening work that
  exceeds the current sprint's budget.
- **Every item** has: source (what discovered it), category,
  short description, and why it is deferred rather than closed now.
- **Removing an item** happens when it lands in a sprint. Link the
  removal to the implementing commit in the commit message.
- Items are not prioritised here; prioritisation happens at v1.1
  planning time using the current PAFM and updated success criteria.

## Entries

### B-001 — Access-token lifetime policy + role-change revocation

**Source:** Security review SR-S0-04 (2026-04-24).
**Category:** Auth hardening.
**Description.** JWT access tokens carry the `cims:role` claim baked
in at issue time. Demoting an admin leaves them with residual
authority until the token expires. Define `AccessMinutes` as a
documented SLA in an ADR and add a revocation path that forces token
refresh on role change.
**Why deferred.** Requires a revocation-store or token-denylist
design; standard JWT implementations accept this window. No
single-incident justification to spend S0 budget on it.

### B-002 — Rate limiting and abuse protection on anonymous endpoints

**Source:** Security review SR-S0-06.
**Category:** Platform hardening.
**Description.** `POST /auth/register`, `POST /auth/login`,
`POST /auth/refresh`, and `POST /organisations` have no rate limit,
CAPTCHA, or email verification. Add ASP.NET Core rate-limiting
middleware, progressive back-off on login failures, CAPTCHA or
email-verification on organisation creation.
**Why deferred.** No hostile traffic observed on the dev deployment;
adding abuse mitigation before a first real customer is premature.
Revisit before first external tenant onboards.

### B-003 — Constant-time register path

**Source:** Security review SR-S0-07.
**Category:** Auth hardening.
**Description.** `AuthService.RegisterAsync` first checks email
existence, then organisation existence. Differing latencies leak a
small email-enumeration oracle. Merge the checks into a single
constant-time flow, or defer to an auth-hardening sprint with other
timing-oracle audits.
**Why deferred.** Low impact — emails are generally non-secret.
Batch with other auth hardening.

### B-004 — Contractual-relationship check at project creation

**Source:** Security review SR-S0-08.
**Category:** Billing / admin.
**Description.** Beyond the tenant-ownership check (SR-S0-02, which
is S0-blocking), create-project has no validation that the calling
organisation is entitled to create projects — contract active,
billing up to date, seat count not exceeded, plan supports the
feature. Belongs to the admin console / billing work.
**Why deferred.** S14 Admin Console is the natural home; premature
here.

### B-005 — Assignee ownership check on `PATCH actions/{id}`

**Source:** `docs/security/role-matrix.md#known-deferred-checks`.
**Category:** Domain authorization.
**Description.** `TaskTeamMember+` is the current floor for editing
an action. A stricter model would require the caller to be the
action's assignee or a `ProjectManager+`. Decide at the sprint that
introduces actions workflow polish (likely S7 Reporting or S14
Admin).
**Why deferred.** Floor is sufficient for v1.0 daily-use. Precision
check is polish, not safety.

### B-006 — Responder verification on `POST rfis/{id}/respond`

**Source:** `docs/security/role-matrix.md#known-deferred-checks`.
**Category:** Domain authorization.
**Description.** Similar to B-005 — current floor is
`TaskTeamMember+`; optimal check is "only the RFI's intended
responder or an `InformationManager+`". Defer to RFI workflow polish.
**Why deferred.** Same as B-005.

### B-007 — Admin scoping on `GET /api/v1/organisations`

**Source:** `docs/security/role-matrix.md#known-deferred-checks`.
**Category:** Admin UI.
**Description.** Endpoint returns all organisations visible through
the tenant query filter. For ordinary users this is their own org;
for `SuperAdmin` this is all orgs. Acceptable by ADR-0003 but likely
evolves when the admin console (S14) is built.
**Why deferred.** Correct today per ADR-0003. Revisit at S14.

### B-008 — Reconcile `Services/Iso19650/Iso19650ReferenceData.cs`
with `Core/Iso19650Codes.cs`

**Source:** CR-001 impact note; memory
`project_roadmap_position.md`.
**Category:** Technical debt from Sessions 1–2.
**Description.** Two divergent reference-data sources for ISO
19650 codes exist. Must be unified at S8 kickoff when the parked
validator is integrated with the naming wizard / MIDP / TIDP flow.
**Why deferred.** S8 is the natural home. Parked per ADR-0008.

### B-009 — SSO / federated IdP support and multi-role-claim-type
handling

**Source:** Security review SR-S0-03 (forward-looking).
**Category:** Auth architecture.
**Description.** `RoleClaimType = "cims:role"` currently maps a
single custom claim to ASP.NET's role concept. If federated SSO is
added, an external IdP may emit `ClaimTypes.Role`. Revisit whether
both claim types should be honoured or whether a claims-transformer
maps external roles to `cims:role`.
**Why deferred.** No SSO customer in v1.0. Revisit before first B2B
customer onboards.

### B-013 — MudBlazor 7 upgrade

**Source:** S0 retrospective notes-for-next-sprint; CR-003 (S1
scope cut).
**Category:** Tech debt / UI framework upgrade.
**Description.** Upgrade MudBlazor 6.20.0 → 7.x. Resolves the
pre-existing `Palette` deprecation warning at
`Components/Layout/MainLayout.razor:82` and removes future risk
on inline-dialog patterns (the silent-no-op bug the S0
retrospective flagged on `<MudDialog @bind-Visible>`).
**Why deferred.** Genuine breaking-change pass (PaletteLight /
PaletteDark split, theme API changes, plus assorted property
renames). Nothing in S1's F.2 scope depends on it; the warning
is informational and CI does not fail on it.

### B-014 — Construction Act payment notice and pay-less notice flows

**Source:** PAFM-SD `.docx` Appendix F.2; CR-003 deferred from S1.
**Category:** S1 fast-follow OR v1.1.
**Description.** Statutory notice flows under the
Housing Grants, Construction and Regeneration Act 1996 — payment
notice issued by paying party, pay-less notice for any reduction.
Statutory deadlines (5 days from due date for payment notice,
7 days before final date for pay-less). PDF generation via the
existing template path used by payment certificates.
**Why deferred.** Depends on T-S1-09 payment certificate
generation landing first; S1 capacity (72h formal, 96h
S0-adjusted) does not fit both with confidence. Bring back as a
fast-follow mini-sprint after S1 close OR roll into S5 Change
Control where the workflow conventions overlap. Compliance
risk: shipping payment certs without these flows means UK
construction users would have to track notices off-system —
acceptable for v1.0 internal pilot, not for an external customer.
Re-evaluate before any external onboarding.

### B-015 — Final account schedule

**Source:** PAFM-SD `.docx` Appendix F.2; CR-003 deferred from S1.
**Category:** Late-stage feature (v1.0 fast-follow OR v1.1).
**Description.** Project-closeout rollup view tying CBS lines,
final variations valuations, retention release, and any pending
commitments into a single agreed-final-account document.
**Why deferred.** Late-stage in the project lifecycle — most
projects do not need it until practical completion. None of the
S0 success-criteria pilot projects are at PC yet. Defer until
the pilot reaches that stage OR until the same pilot demands it.

### B-016 — Variations workflow: full 6-state machine

**Source:** PAFM-SD `.docx` Appendix F.2; CR-003 trimmed for S1.
**Category:** Process polish (v1.1).
**Description.** Restore the full F.2 variations workflow:
raise → **assess** → approve(delegation) → **instruct** →
**value** → **agree** (intermediate states bolded). T-S1-08 in
S1 ships only the core raise → approve → reject states; this
backlog item adds the four intermediate states.
**Why deferred.** The intermediate states are mostly
process-tracking on top of the core state machine. Approve /
reject is the load-bearing decision; assess / instruct / value /
agree are auditable positions in the workflow but do not change
the financial outcome. Acceptable to defer for v1.0 pilot use.

**Why deferred.** Anti-scope. AI-assisted *development* is fine and
in use; AI-assisted *product* is v1.1.
