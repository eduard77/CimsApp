## Findings

### SR-S0-01 — Anonymous registration allows cross-tenant user creation **(Critical)**

**Location.** `CimsApp/Controllers/Controllers.cs:32-34`,
`CimsApp/DTOs/Dtos.cs:5`, `CimsApp/Services/Services.cs:23-33`.

**Observation.** `RegisterRequest` is
`(string Email, string Password, string FirstName, string LastName,
string? JobTitle, Guid OrganisationId)`. `AuthController.Register` is
`[AllowAnonymous]`. `AuthService.RegisterAsync` uses
`req.OrganisationId` directly with no verification beyond "the org
exists". There is no invitation token, no admin approval, no
email-domain match, and no link between the caller and the target
organisation.

**Impact.** A remote anonymous attacker who knows any organisation's
Guid (easily obtainable: any authenticated user can list all
organisations via `GET /api/v1/organisations`, Guids may also leak
through support channels, invoices, URLs, or error messages) can
`POST /api/v1/auth/register` with `organisationId: <target>`. The
created `User` row carries the target `OrganisationId`. After
calling `/auth/login`, the attacker's JWT carries the `cims:org`
claim for the target tenant. Every `CimsDbContext` query filter —
shipped in commit `3c95027` to implement T-S0-03 and close PAFM
Appendix F.1 row-level isolation — resolves against the attacker's
injected tenant claim. The attacker can list and read every
tenant-scoped entity (Projects, Documents, RFIs, Actions, AuditLogs,
etc.) of the target organisation.

**Why this matters to Sprint 0.** PAFM Appendix F.1 DoD item
"Users are scoped to organisations" is technically satisfied by the
`User.OrganisationId` column but functionally bypassed because the
scoping is attacker-chosen at registration. The entire query-filter
investment (T-S0-02, T-S0-03, T-S0-04a, T-S0-07) is rendered
ineffective at the boundary. The Sprint 0 DoD cannot honestly be
marked complete with this open.

**Recommendation.** Before sprint close, choose one:

1. **Closed registration.** Remove the `OrganisationId` field from
   `RegisterRequest`. Registration happens only via an invitation
   token generated by an OrgAdmin. The token encodes the target
   organisation and is validated server-side. Most construction SaaS
   products follow this model (Procore, Aconex, Asite).
2. **Domain-bound registration.** Organisation has an allowed-domains
   list; `Register` only succeeds if `email` domain matches. Lower
   assurance than invite tokens; still closes the anonymous-Guid
   bypass.
3. **Approval queue.** `Register` creates a pending-user row; an
   OrgAdmin must approve before the user can log in. More ceremony;
   may match PAFM-SD "no tacit approvals" language.

This is a construction-domain decision (how does CIMS provision site
team members?). I cannot pick silently. Raising as decision point
SR-S0-01 for Eduard.

### SR-S0-02 — Project creation accepts attacker-controlled `AppointingPartyId` **(High)**

**Location.** `CimsApp/Services/Services.cs:129-136`,
`CimsApp/DTOs/Dtos.cs:12`,
`CimsApp/Data/CimsDbContext.cs:165`.

**Observation.** `ProjectsService.CreateAsync` writes
`AppointingPartyId = req.AppointingPartyId` verbatim. The tenant
query filter for `Project` is
`p.AppointingPartyId == _tenant.OrganisationId`. Nothing in the
create path verifies that the caller's organisation has a legitimate
relationship with the target appointing party.

**Impact.** An `OrgAdmin` of organisation A (now that T-S0-08 gates
`POST /projects` on `OrgAdmin/SuperAdmin`) can create a project with
`AppointingPartyId = orgB.Id`. The new project lands in orgB's
query-filter scope, disappearing from orgA's view. OrgB now sees a
project they did not create and cannot attribute. Downstream audit
trail, member assignments, and document creation could all accrue
under the attacker's control.

**Why it's High not Critical.** Requires an authenticated
`OrgAdmin`, so the attacker is already inside the trust boundary.
Still a cross-tenant write primitive.

**Recommendation.** At create-time, require **one of**:

1. `req.AppointingPartyId == caller.OrganisationId` (caller's org
   is the appointing party — they are the client side of the
   contract). OR
2. A `ProjectAppointment` row already exists linking
   `caller.OrganisationId` to `req.AppointingPartyId` with
   appropriate role (caller's org is appointed by the target). OR
3. Caller is `SuperAdmin` (platform-level bypass, audited).

Construction domain check: UK construction typically has the
appointing party as the client-side organisation and the appointed
party as the delivery consultant / contractor. Decide whether the
appointing party is always the caller's own org (common for in-house
PMO tools) or whether another org can appoint the caller (B2B
platform pattern).

### SR-S0-03 — `RoleClaimType` override is a footgun **(Medium)**

**Location.** `CimsApp/Program.cs:34`.

**Observation.** Commit `f41c410` added
`RoleClaimType = HttpTenantContext.GlobalRoleClaimType`. This tells
ASP.NET authorization to read `"cims:role"` as the role claim
instead of the default `ClaimTypes.Role`.

**Impact.** Any future code that calls `User.IsInRole(...)`,
`User.FindFirst(ClaimTypes.Role)`, or reads `ClaimsIdentity.Role`
will now silently operate against `"cims:role"` only. If a feature
later adds a second role source (e.g., a federated-SSO IdP emitting
`ClaimTypes.Role` claims), those claims will be ignored by
authorization. No runtime error signals the drop.

**Recommendation.** Add an inline comment in `Program.cs` explaining
the override and linking to ADR-0010. If SSO or additional identity
providers land in a later sprint, revisit whether both claim types
should be honoured.

### SR-S0-04 — Stale role claim after demotion **(Medium)**

**Location.** `CimsApp/Services/Services.cs:78-93`, token generation.

**Observation.** JWT access token carries the `cims:role` claim
baked in at issue time. Revoking a user's `GlobalRole` (e.g.,
downgrading OrgAdmin to Viewer) does not invalidate the outstanding
access token. Until the token expires or a refresh occurs, the
downgraded user retains admin-role authorization.

**Impact.** Duration equal to `AccessMinutes` (not verified here;
configured in `Jwt:AccessMinutes`). If set to e.g. 60 minutes, a
demoted OrgAdmin has up to 60 minutes of residual authority. Can be
exploited during suspected-compromise response windows.

**Recommendation.** Keep `AccessMinutes` short (15–30 min is
conventional). On role change, optionally issue a revocation event
that forces refresh. Document `AccessMinutes` as an SLA in a future
security ADR.

### SR-S0-05 — Orphan ProjectMember rows **(Low)**

**Location.** `CimsApp/Services/Services.cs:138-144`.

**Observation.** `AddMemberAsync` inserts a `ProjectMember` without
verifying that the target user's `OrganisationId` matches the
project's `AppointingPartyId`. The User query filter prevents
looking up foreign-tenant users normally, but `AddMemberAsync`
writes without reading, so the write succeeds.

**Impact.** Tenant query filters on ProjectMember do hide the
orphan at read time (the join filter discards cross-tenant rows),
but the DB now has rows that silently disagree with the model. Also
a correctness risk for future reports.

**Recommendation.** Before insert, verify the user's
`OrganisationId` equals the project's `AppointingPartyId` (or a
sibling check per SR-S0-02's resolution). Throw `BadRequestException`
otherwise.

### SR-S0-06 — No abuse protection on anonymous endpoints **(Low)**

**Location.** `POST /api/v1/auth/register`,
`POST /api/v1/auth/login`, `POST /api/v1/auth/refresh`,
`POST /api/v1/organisations`.

**Observation.** No rate limiting, no CAPTCHA, no email-verification
flow on any anonymous endpoint. Organisation creation is fully open.

**Impact.** Spam org creation, password-spray login attempts,
refresh-token-cycling probes. None sprint-0-blocking.

**Recommendation.** Defer to a later hardening sprint; track in
v1.1 backlog with specific controls: (a) ASP.NET rate-limiting
middleware on auth endpoints, (b) CAPTCHA or email verification on
`POST /organisations`, (c) progressive delay on login failures.

### SR-S0-07 — Timing oracle on email existence **(Low)**

**Location.** `CimsApp/Services/Services.cs:26-28`.

**Observation.** `RegisterAsync` first checks email existence
(`db.Users.IgnoreQueryFilters().AnyAsync(...)`), then looks up
the organisation. A registered email with wrong org returns a
different latency profile than an unknown email. Distinguishable
via high-resolution timing.

**Impact.** Low. Emails are often low-sensitivity
(LinkedIn-discoverable). Still worth fixing when auth hardening
comes up.

**Recommendation.** Either merge the two checks into a single
constant-time path, or defer to an auth hardening sprint.

### SR-S0-08 — No contractual relationship check on project creation **(Info)**

**Location.** `CimsApp/Services/Services.cs:129-136`.

**Observation.** Adjacent to SR-S0-02: even if the
`AppointingPartyId` points to the caller's own organisation,
there's no check that the caller's `Organisation` is entitled to
create projects (contract active, billing up to date, etc.). Pure
info — belongs to the Billing / Admin sprint, not S0.
