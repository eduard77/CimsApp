## Retrospective

### What went well

- **CR-004 surfaced and resolved on day 1 — third sprint in a
  row where Ch 17 worked as designed.** The kickoff F.3 backfill
  made the capacity gap visible (98h vs 72h formal); Option A
  scope cut was raised, accepted, and recorded on the same day
  rather than absorbed silently. Same shape as CR-002 (S0
  hardening discovery) and CR-003 (S1 capacity cut). The
  discipline manual continues to work.
- **The S0 / S1 patterns transferred cleanly to S2.** Every
  pattern landed in earlier sprints — tenant query filter shape,
  audit-twin atomicity (PR #33), role-matrix update protocol,
  three-commit-per-task cadence (entity / migration / tests),
  EF migration NoAction-everywhere — applied without
  modification. The cost-domain entities from S1 were the
  template for the risk-domain entities, and the existing
  `CostBreakdownItemFilterTests` was the template for the
  RBS / Risk / RiskDrawdown filter tests.
- **`Core/MonteCarlo.cs` followed the `Core/Evm.cs` purity
  stance.** No IO, no DB, no DI. Caller supplies the `Random`
  for determinism control (production uses `Random.Shared`,
  tests use seeded `Random(42)`). PMBOK fixture-based tests
  (Triangular mean = (B+M+W)/3, PERT mean = (B+4M+W)/6) catch
  sign / divisor errors that decimal-comparison tests would
  miss.
- **The audit-twin pattern carries fluent semantics.** Every
  risk mutation lands two audit signals (per-row from the
  interceptor + structured event from the service) committed
  atomically. Eight new structured action names this sprint —
  `risk.created`, `.updated`, `.closed`, `.qualitative_assessed`,
  `.quantitative_assessed`, `.drawdown_recorded` — all follow
  the same shape as S1's eleven cost-domain events.
- **Risk-score threshold flexibility was preserved.** The
  matrix returns numeric scores 1..25; threshold mapping
  (low / medium / high) intentionally not baked in — that's a
  per-tenant Admin Console setting in S14. Future configurability
  protected by NOT making a default mapping load-bearing now.

### What did not go well

- **Sprint cadence remains anomalous.** Same problem as S0 / S1:
  a 75h-trimmed sprint completed same-day. The retrospective
  says "this is not a sustainable baseline" for the third time.
  The honest read: AI-paired work on well-specified incremental
  features against a mature codebase compresses far below
  formal capacity, but the velocity is bounded by how
  specification-ready the next module is. S3 Stakeholder &
  Communications has more interaction-design judgement and may
  not compress the same way; S4 Schedule & Programme has CPM
  algorithms with real edge-cases (cycles, lag/lead semantics,
  resource levelling) that the existing patterns don't help
  with.
- **One pattern-deviation tempted me mid-sprint.** During
  T-S2-09 RiskDrawdown design, the cross-module link to
  Commitment FK (B-030) felt like a small extra step — would
  have wired up cleanly. Caught before any code, but it shows
  the discipline weakness: when a deferral feels easy at the
  moment of decision, it requires conscious resistance to
  re-open the change record. This is exactly what CR-004
  envisioned protecting against; the protection worked but
  only because of the explicit reminder in `docs/sprint-log/s2.md`.
  Action: the kickoff template's deferral list is doing real
  load-bearing work — keep visible during execution, not just
  at kickoff.
- **The 75h sprint estimate was off by ~10×.** Like S0 (49h
  estimated, ~2 days actual) and S1 (96h post-cut, ~2 days
  actual), the S2 estimate massively over-shot the realised
  effort. Three sprints of evidence; the formal capacity
  calculation does not predict actual effort under AI-paired
  development. Action for next kickoff: continue producing the
  formal estimate (it disciplines decomposition and surfaces
  capacity gaps) but stop treating it as a load-bearing
  prediction. Track velocity differently — perhaps "modules
  closed per session" rather than "hours per task" — once a
  baseline of three sprints is converted from anomaly to
  expected behaviour.

### What surprised me

- **Monte Carlo arrived at 18 pure-function tests in one go.**
  The 3-distribution sampler + percentile aggregator + simulation
  runner came in at ~150 lines of math + 18 tests, well below
  the 9h estimate. The textbook PMBOK formulas (Triangular mean,
  PERT mean) are exact targets for behavioural assertions —
  no domain ambiguity to litigate.
- **The Risk → RiskCategory FK is the v1.1 RBS-template hook.**
  Per-project RBS in v1.0 mirrors per-project CBS from S1.
  Future template-seeded RBS (org-level standards copied to
  new projects) is a natural extension; the current schema
  doesn't preclude it.
- **Beta and PERT collapsed to one sampler in v1.0.** Per the
  `MonteCarlo.cs` doc-comment, the normal-approximation Beta
  sampler is a placeholder; an exact Marsaglia-Tsang Gamma-ratio
  sampler is a v1.1 candidate. Surprising how little the
  approximation matters at 1000+ iterations — well within the
  ±5% bands the tests assert.

### What I will try differently next sprint

- **Revisit the formal-capacity calculation.** Three sprints in,
  the 6h × working days × 80% formula is a poor predictor of
  actual effort under the current working pattern. Don't drop
  it (it disciplines task decomposition), but stop framing
  CR-XXX cuts as "fitting capacity" and re-frame as "scope
  discipline" — the cuts are about which features genuinely
  belong in v1.0, not about what fits in 72 hours.
- **Run security-review at midpoint, not just close.** S0's
  retro flagged this and S1 / S2 didn't pick it up because
  there was no midpoint. With sprints completing same-day,
  the security-review pass is best run as a separate "post-
  sprint pre-merge" step before the PR — same shape as the
  S1 SQL-Server smoke walk that found four production bugs.
  Not done in S2; queue for S3.
- **Pre-flight the next module with an Explore agent before
  kickoff.** The S0 / S1 / S2 pattern has been: write the
  kickoff, then immediately discover scope-cut needs. Running
  an Explore over the F.X module spec + adjacent codebase
  before writing the kickoff might catch capacity issues
  earlier — at the spec-extraction step rather than the
  decomposition step.
