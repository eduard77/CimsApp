## Retrospective

### What went well

- **The S5-retro cadence prediction was tested and held.** S5
  predicted that Approve→Variation pattern would re-apply for
  Award→Contract; "workflow-shaped domain again — predict
  same-session compression". Result: it compressed. F.7's 5
  bullets cover 8 entities + 7 services + 2 state machines +
  1 weighted aggregator + 1 cross-service spawn — all delivered
  end-to-end in one session. The "novel decision = cadence-breaker"
  hypothesis from S4 retro now has 3 falsifications (S4, S5, S6).
  Real cadence-breakers remain explicit predictions for S15
  (Mobile UI), S17 (Security pen test), and B-027 CI hardening
  sprint.
- **The pattern reuse rate is now extreme.** Concrete reuse
  count from S6:
  - `ChangeWorkflow` shape (S5) → re-applied as
    `TenderPackageWorkflow` (T-S6-03) and
    `CompensationEventWorkflow` (T-S6-08). Same 3-dictionary
    structure, same `HasMinimumRole` reuse, same `IsTerminal`
    helper.
  - `RiskMatrix` / `StakeholderMatrix` pure-function shape (S2/S3)
    → re-applied as `EvaluationMatrix` (T-S6-05). Same shape:
    pure, no IO, weight-sum invariant + epsilon validation.
  - S5 ChangeRequest.ApproveAsync atomic spawn → re-applied as
    TenderPackage.AwardAsync (T-S6-06). Same transactional
    SaveChanges + multi-event audit chain.
  - V1 Variation + ChangeRequest CR-NNNN auto-numbering → re-used
    for TP-NNNN, CON-NNNN, CE-NNNN. The unique
    (ProjectId, Number) index racing pattern is now stable.
  Reuse means new entity work is mostly just typing.
- **B-NNN inline discipline held cleanly.** S5-retro action:
  "every v1.1 candidate decision adds a B-NNN entry in the same
  commit as the decision, not as a closing audit." S6 Top-3
  risks pre-flagged B-048..B-054 at kickoff; T-S6-08 landed
  B-048 / B-049 / B-050 in the same commit as the
  CompensationEvent feat. Closing audit was 0 entries.
  T-S5-08-style retroactive sweep: not needed.
- **Smoke walk ran as the FIRST close action per the S5-retro
  re-shaped rule.** S5 retro landed the rule but didn't
  itself follow it (the retro was written before the smoke
  walk). S6 followed the rule cleanly: smoke walk first
  (22 steps green, no production bugs), then DoD ticks, then
  this retro. Discipline change held.
- **Pragmatic test-pattern departure was well-judged.** T-S6-09
  filter sweep introduced 8 new entities all using the same
  `Project.AppointingPartyId` filter shape. Eight near-identical
  XxxFilterTests files would have added high duplication for low
  marginal coverage. Decision: consolidate into one file with
  2 representative entities (top + bottom of the chain) +
  layer the model-level + service-level cross-tenant tests
  underneath. Test count delta dropped from a hypothetical +16
  to actual +3 — the right shape. The discipline manual doesn't
  prescribe per-entity files; consistency-for-its-own-sake
  isn't a goal.

### What did not go well

- **Sprint cadence is now boring as a finding.** S0..S6 same-day
  compression with the only inflection being S5's overnight rest.
  Seven sprints of evidence; the next thing worth flagging is
  not "cadence broke today" but "cadence finally broke at S15".
  The retros are starting to repeat themselves on this point —
  worth retiring the cadence-breaker prediction as a structured
  retro section unless / until it actually breaks. From S7
  onwards, cadence reporting compresses to a single line
  ("substantive code: 1 session") unless something changes.
- **The pattern-reuse rate is so high that S7+ may not need
  state-machine implementation discussion as a separate task.**
  `Core/TenderPackageWorkflow.cs` and
  `Core/CompensationEventWorkflow.cs` both took ~10 minutes each.
  Both passed all fixture tests first try. The interesting work
  in S6 was the **decisions** (BSA categorisation as tag vs.
  inference; multi-stage vs single-stage tendering; weight-sum
  validation timing) — not the implementation. Future kickoffs
  should weight decisions heavier than implementation in time
  estimates.
- **Lingering scheduled remote agent still un-deleted** —
  `trig_01CbdDcec2Q2GQK8oidWjm3o`. Six retros now. No execution
  risk. Manual delete via the web UI required. Continuing to
  carry forward feels increasingly absurd; explicit one-line
  rule from S7 onwards: "only flag in retro if a status change
  has occurred."

### What surprised me

- **All 39 CompensationEvent tests passed first try, including
  the NEC4 clause 61.4 Notified→Rejected branch.** I'd expected
  the dual-path rejection logic (Notified→Rejected vs
  Quoted→Rejected) to surface a state-machine edge case, but
  the dictionary-of-(from, to)→role pattern handled it without
  modification. The test fixtures explicitly cover both paths
  and both were green.
- **EvaluationMatrix's weight-sum-at-matrix-calc-time decision
  was the right call.** Validating at criterion-write time
  would have rejected weights legitimately mid-edit (e.g.
  setting Price=0.6 before adding Quality=0.4 momentarily
  fails Σ=1.0). Validating at matrix-calc time allows the
  workflow to stabilise; the fixture test
  `Compute_with_invalid_weight_sum_still_returns_per_tender_results`
  documents the IsValid=false flag for the UI to show
  "weights don't sum to 1.0" warning. Cleaner than a hard
  rejection mid-edit.
- **Test count crossed 700 then 750 in the same sprint.** S6
  delta of +149 is the second-largest after S4's +133. The
  v1.0 pilot target of ~1000 tests is now likely reachable
  by S7 or S8 on the current trajectory.

### What I will try differently next sprint

- **Drop "cadence-breaker prediction" from the retro structure
  unless it actually breaks.** S7 onwards: replace the cadence
  paragraph with a one-line "substantive code: 1 session"
  status. Only break it out into a full retro section if a
  real inflection happens. Saves retro space for the things
  that actually change between sprints.
- **Move state-machine implementation to "near-zero estimate"
  in kickoffs.** Both Core/TenderPackageWorkflow.cs and
  Core/CompensationEventWorkflow.cs took ~10 minutes each
  (+ tests). The kickoff estimates of 3-6h per state machine
  task are 10-30× the actual time. Future kickoffs should
  budget state machines at 1h flat including tests.
- **Explicitly stop carrying the disabled scheduled agent
  through retros after S7.** The "delete it manually" flag
  has been in 6 retros. From S7: only re-mention if status
  has changed. Otherwise it's noise.
- **Audit the v1.1 backlog at S7 kickoff for any items that
  could land as fast-followers.** With 54 entries (B-001..B-054)
  in the backlog, some may be small enough to fit alongside
  the next sprint's main work. Specifically: B-035 (CR
  structural impact links) might be a 2-3h add inside S7.
