## Retrospective

### What went well

- **Two real bugs caught before they hit master** — exactly
  the value B-027 was designed to deliver. The first push
  surfaced the `Organisation.Code` MaxLength(10) overflow
  (smoke script generated 15-char codes via `$RANDOM-$$`
  expansion). The second push surfaced the
  RegisterAsync-returns-UserSummaryDto-not-AuthResponse bug
  in the smoke script's step-2 assumption. Neither would
  have been caught by unit tests; both were caught in
  ~3 min CI iteration cycles, fixed in the same session.
  **The CI hardening sprint paid for itself on its own
  validation push.**
- **Migration round-trip job exceeded expectations.** The
  kickoff Top-3 risk #3 predicted "1-3 migrations have
  broken Downs"; budget allocated up to 1h for fixes.
  Result: **all 60+ migrations across S0..S6 round-trip
  cleanly first try.** Honest finding: EF Core's auto-
  generated Down() methods are reliable in practice; the
  20+ commits where Eduard ran `dotnet ef migrations add
  ...` produced consistent reversibility. The
  ~5-line YAML cost catches a future class of bug for
  near-zero ongoing effort.
- **The cadence-prediction caveat held.** S6 retro retired
  the cadence-breaker section "unless it actually breaks";
  S8 was specifically called out as plausibly different
  (infrastructure-shaped, longer iteration cycles). Honest
  prediction at kickoff was "1 session if Option A is
  straightforward, 2 if not". Outcome: **1 session**, with
  the 2 mid-task fixes treated as part of the normal CI-
  validation iteration. Worth recording as a calibration
  data point: infrastructure work is NOT inherently slower
  than feature work when the underlying tooling
  (`mssql/server` Docker, GitHub Actions service containers)
  is well-established.
- **Build+test job warnings-as-errors held the 0-warnings
  bar.** Pre-S8 the CI was lenient on warnings; post-S8
  any new warning is a build failure. The first CI push
  with `-p:TreatWarningsAsErrors=true` was green — meaning
  the existing codebase across S0..S6 is genuinely
  warning-free, not "warnings nobody bothered to fix
  yet". Confirmation that the discipline-manual code
  hygiene held through 7 sprints of feature work.
- **B-NNN inline discipline now feels routine.** B-061..
  B-064 landed in T-S8-06's same commit, no closing
  audit. Third sprint in a row to honour the rule (S6,
  S7, S8); the S5-retro discipline change is genuinely
  embedded.

### What did not go well

- **Two smoke-script bugs in a row was avoidable with more
  careful script review.** The `$RANDOM-$$` overflow
  should have been caught at write time by counting chars
  + cross-referencing the `Organisation.Code` MaxLength
  attribute. The register-returns-UserSummaryDto
  assumption should have been caught by reading the
  `RegisterAsync` signature in the service layer. Both
  were trivially-discoverable from the codebase. The fact
  that CI caught them is a feature; the fact that the
  script had them is a soft "could have been more careful"
  — but the cost was 2 × ~3min iteration loops, not a
  significant rework. **Lesson:** for CI-shaped work, the
  cycle "write script → first CI run → find real bugs" is
  *expected*, not a process failure. Don't over-correct on
  this finding.
- **The sprint-level DoD checklist required adaptation.**
  "All tests for implemented features pass" reads
  awkwardly when the implementation has no in-process
  tests (the smoke job IS the test). The sprint log
  notes this with a parenthetical, but a future
  infrastructure sprint might want a separate sprint-
  specific DoD section that supersedes the generic one.
  S8 invented this pattern (sprint-specific DoD against
  B-027); future B-NNN sprints can re-use it.

### What surprised me

- **`dotnet ef database update 0` is reliable.** I had
  half-expected the Down() round-trip to surface at
  least one broken migration. Zero. The EF Core auto-
  generation pipeline appears genuinely solid for the
  schema shapes CIMS uses (pure CREATE TABLE / ADD
  COLUMN / ADD INDEX / FK constraints — no
  hand-written DROP/CASCADE oddities). Reframes my
  internal trust level for the migration tooling.
- **GitHub Actions service containers + 90s health
  wait were enough.** The kickoff Top-3 risk #1
  ("readiness probe timing fragile") never fired —
  `mssql/server` consistently reported healthy in
  <5s in all three CI runs. The belt-and-braces wait
  loop never reached iteration 2. Worth noting that
  the standard idiom (service-level `--health-cmd`
  with retries) plus a workflow-step poll is genuinely
  belt-and-braces, not over-engineering — if either
  layer fails the other catches it.
- **The "smoke walk IS the CI run" framing was new
  but felt right.** Earlier sprints had a literal
  `smoke-sN.ps1` script run against a local SQL Server
  as the FIRST close action; S8's smoke walk was a
  green CI run on the open PR. The S7 retro had
  flagged "S8 default reverts to a literal script" —
  but for S8 specifically, the CI workflow itself IS
  the smoke script (just running on GitHub's
  infrastructure instead of locally). The discipline
  rule needs a small refinement: from S9 onwards, the
  smoke walk = "green CI on the open PR" by default,
  with the local PowerShell script as a
  user-discretion add-on for sprints where local
  pre-CI verification helps the iteration loop.

### What I will try differently next sprint

- **Re-frame the smoke-walk discipline rule.** S5-retro:
  "smoke walk runs as the FIRST close action". Pre-S8:
  smoke walk = run smoke-sN.ps1 against local SQL
  Server. Post-S8: smoke walk = "first green CI run on
  the open PR" — the CI workflow added in S8 IS the
  smoke harness, and it runs on every push to the
  sprint branch. Local PowerShell smoke scripts become
  a user-discretion tool for tightening the iteration
  loop on sprints where the CI cycle is too long for
  the bug class. Update the discipline manual / next
  sprint kickoff to reflect this.
- **Set up master branch protection AS PART OF the S8
  PR merge.** `docs/security/ci-required-checks.md`
  documents the manual UI step; once the S8 PR merges
  and the three CI jobs (Build and test / SQL Server
  smoke / Migration round-trip) have run on `master`,
  add them as required checks on the `master` branch.
  Note: requires admin access to the repo; cannot be
  done from CI itself per ADR-0015.
- **Decide S9 scope at the next kickoff.** S6/S7 retros
  flagged "B-027 OR module work; re-evaluate"; S8 took
  B-027. S9 candidates:
  - **F.9 / next module per the PAFM-SD roadmap** —
    return to feature work now that CI is hardened.
  - **B-061 / B-062 / B-063 — second pass of CI
    hardening.** Coverage / perf / security gates each
    add a layer; could be one S9 sprint or pushed to
    a later "v1.1 follow-up" sprint.
  - **B-NNN backlog audit + pruning pass.** Backlog is
    now at 64 entries (B-001..B-064). S6/S7 both flagged
    audit pass at next kickoff; S8 didn't have time
    (legitimately — kickoff was tight). S9 has the
    capacity if it's chosen as the lighter sprint.
  Lean: F.9 module work next (return to features) with
  audit pass + B-061..B-063 deferred to the post-
  S9 / pre-S10 retro decision.
