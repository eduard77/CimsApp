## Retrospective

### What went well

- **Triple Option A kickoff held cleanly.** SMS deferral, in-memory
  queue, simple AlertRule shape — all three honoured through to
  close, no scope creep, no mid-sprint pressure to revisit. The
  triple-decision pattern is now empirically 4-for-4 (S10 / S11 /
  S12 / S14).
- **Pure-logic separation paid off.** `ThresholdRule.IsBreached` +
  `IsInCooldown` + `AlertMetricComputer.ComputeAsync` were
  trivially unit-testable (8 of 13 evaluator tests are pure
  decision-table). The hosted service reduced to wiring +
  fan-out — the fragile bit of background work — without
  decision logic baked in. Pattern to carry: when a background
  service has any non-trivial decision, factor that decision
  into pure helpers first.
- **Background-service tenancy hazard managed cleanly.** The
  Top-3 risk #3 prediction (multi-tenant evaluator) was right —
  the service runs outside request scope, so it explicitly
  bypasses the global query filter via `IgnoreQueryFilters()`
  on every read. Hostile concurrent tenants would still see
  their own data only because every query carries a literal
  `ProjectId` — no implicit filter relied on.
- **No new NuGet packages.** SMTP via `System.Net.Mail.SmtpClient`
  (BCL); SignalR + `IHostedService` part of ASP.NET Core. Avoided
  the Ch 24.6 24-hour wait — sprint stayed single-session.

### What did not go well

- **Test fixture noise on T-S14-04.** First build attempt failed
  on `Risk.Number` and `Commitment.CounterpartyName` — both
  guessed property names that don't exist on the v1.0 entities
  (Risk has no Number, Commitment is `Counterparty`). Should
  have skimmed `Entities.cs` before writing test-data builders.
  Caught at build time; ~3 minutes lost. Generalisable lesson:
  **before writing test data builders for entities you haven't
  touched recently, grep / read the entity definition.**
- **+27 tests is mid-density.** Honest sprint shape: SignalR
  hub-side behaviour (group join / leave / push) was reflected
  rather than tested; the FakeHubContext stand-in records sends
  but doesn't exercise the real connection / authentication
  path. End-to-end SignalR auth (the `?access_token=` query-param
  fallback) would need an integration test against a running
  hub; v1.0 ships with the wiring + the unit-tested fan-out.
  Pre-pilot QA may want this — flag as a v1.1 fast-follower
  test rather than a B-NNN.

### What surprised me

- **Bounded channel `DropOldest` mode is exactly the v1.0 shape.**
  Capacity 1024 is generous (per-tick fan-out is rarely > 10
  emails); dropping the oldest if a real burst hits is honest
  given the kickoff Decision 2 ("restart loses pending mail
  by design"). No reason for the bound itself; just defending
  the process from runaway producers.
- **`EmailDispatcherHostedService` per-iteration scope is a
  pattern.** Resolving `IEmailSender` via
  `IServiceScopeFactory.CreateScope()` per message — rather
  than capturing it in the singleton's constructor — is the
  right shape any time a scoped dependency feeds a hosted
  service. The threshold evaluator uses the same shape. Worth
  carrying forward for future background-service work.

### What I will try differently next sprint

- **Read entity definitions before writing test fixtures.** The
  T-S14-04 build break was a 3-minute lesson in over-trusting
  prior-session memory. `Entities.cs` is 2400 lines but the grep
  is fast.
- **S15 scope decision.** Default lean per the post-S8 renumbered
  roadmap is **F.15 Search & Discovery** — global search across
  documents, RFIs, actions, risks, etc. Likely needs an ADR on
  search backend (in-DB LIKE for v1.0 → Lucene/Elasticsearch
  for v1.1?). Non-statutory, infrastructure-shaped — same
  domain shape as S14.
