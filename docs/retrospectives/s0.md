## Retrospective

### What went well

- **Discipline held when the security review surfaced a critical
  gap mid-sprint.** SR-S0-01 (anonymous registration accepting an
  attacker-supplied `OrganisationId`) was a tenant-isolation
  bypass that would have invalidated the entire query-filter
  investment. Rather than waive it or punt to v1.1, the gap was
  formally raised as CR-002 against the existing F.1 DoD and
  closed within S0 via T-S0-11. PAFM-SD Ch 17 worked as designed.
- **Two-tier role authorization (ADR-0010) was the right call.**
  The spec literally said "add `[Authorize(Roles=…)]` everywhere",
  but the real role model is per-project. Recognising the
  attribute-vs-imperative split early prevented an attempt to
  flatten per-project membership into the JWT, which would have
  been wrong and unbounded in size.
- **PAFM-SD Ch 24.6 24-hour cooling-off was honoured rather than
  waived.** ADR-0009 proposed `EF Core InMemory 8.0.2` on
  2026-04-24 with the wait clock starting; T-S0-04 / T-S0-06b
  landed only after the wait elapsed at 2026-04-25T12:42Z. The
  rule was easy to bypass; it was kept.
- **Behavioural tests, not just model-level checks.**
  `CimsDbContextTenantFilterTests` proves the filter is
  registered; `CimsDbContextIsolationTests` proves it actually
  scopes reads at runtime, with two-tenant fixtures and
  IgnoreQueryFilters cross-checks.
- **Audit-trail integrity tests cover what could go wrong.**
  `AuditInterceptorBehaviourTests` includes the recursion check
  (a manual `AuditLog` insert must not produce an
  audit-of-the-audit) and the anonymous-context skip — both
  failure modes that would otherwise only surface in production.

### What did not go well

- **Two false starts on the EF migration for `Invitation`.** The
  first attempt used `dotnet ef migrations add --no-build` which
  reused a stale assembly and produced an empty Up/Down body. The
  second attempt regenerated correctly but failed to apply with a
  SQL Server "may cause cycles or multiple cascade paths" error
  because the Cascade-on-Org plus SetNull-on-User combination
  tripped SQL Server's static check. Resolved by aligning all
  three FKs to `NoAction`, matching the dominant pattern
  elsewhere in the DbContext, and regenerating cleanly. Cost
  ~30 minutes and one rolled-back commit (`0fcff2a` superseded by
  `b528897`).
- **The "New Project" button bug took longer than it should have.**
  Three layers: `<Routes />` was rendered without `@rendermode`
  so the entire app was static SSR (fixed in `9b6be43` /
  `045e0f5`); the bare identifier `InteractiveServer` did not
  resolve without a `using static` (fixed by qualifying to
  `RenderMode.InteractiveServer`); and the inline
  `<MudDialog @bind-Visible>` pattern silently failed to render
  even after interactivity worked (fixed by replacing with an
  `@if (_showCreate)` + `<MudOverlay>` modal in `5c1435c`).
  Diagnosing required a full instrumentation pass (`12db136`)
  to prove which layer was broken. The MudBlazor 6 inline-dialog
  mismatch in particular cost time that more current MudBlazor
  versions would have avoided.

### What surprised me

- **Velocity.** A 49h-estimated 3-week sprint completed in ~2
  working days. Two factors: heavy AI pairing carrying the
  routine work, and the codebase already running before S0 began
  (PAFM-SD Ch 0's "starting line not zero" reality). This is not
  a baseline. Later sprints with novel domain modelling (Cost &
  Commercial S1, Risk S2, Schedule S4) will not compress the
  same way.
- **The security review found two genuine criticals that the
  task-list-driven approach missed.** T-S0-08 was supposed to be
  the role audit, and it caught what it was scoped to. But
  SR-S0-01 (registration) and SR-S0-02 (project AppointingPartyId)
  were tenant-isolation gaps adjacent to S0's scope that no task
  in the original decomposition pointed at. Conclusion: an
  out-of-band security pass before sprint close is worth doing
  every sprint, not just S17.
- **MudBlazor 6 inline `<MudDialog>` silently no-ops.** State
  flipped server-side, the page was interactive, the providers
  were registered — and the dialog still did not render. No
  exception, no log line, nothing in the network tab. The
  diagnostic instrumentation that proved this came out as part
  of T-S0-10 cleanup; the lesson stays.

### What I will try differently next sprint

- **Run `/security-review` (or its hand-equivalent) at mid-sprint,
  not at the end.** Today's review surfaced critical gaps that
  reshaped the remaining sprint scope. Catching them at sprint
  midpoint instead of end-of-day-1 would have been better.
- **For any new EF entity, generate the migration immediately and
  apply to a local DB before writing service / controller code on
  top of it.** SQL Server cascade-path issues do not show up at
  `dotnet build` time; they only surface on `database update`.
  Cheaper to find on day 0 than after three commits of dependent
  service code.
- **Watch out for unqualified Razor identifiers.** `@rendermode="InteractiveServer"`
  failed to resolve because `RenderMode` was not in `using static`
  scope. `@RenderMode.InteractiveServer` is the safer default
  pattern. Putting the qualifier in muscle memory now saves time
  in S15 (Mobile Views) where Razor work intensifies.
- **Run the test suite end-to-end before opening any new editor
  tab.** Several intermediate commits this sprint built but
  weren't tested — fine because the changes were docs or trivial,
  but a habit of `dotnet test` before context-switch will catch
  cascade breakage faster.
