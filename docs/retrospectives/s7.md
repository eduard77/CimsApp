## Retrospective

### What went well

- **Read-shape sprint produced first cadence shape change
  that wasn't a break.** S6 retro asked: "does the same-
  session compression hold for read-heavy aggregation
  shape?" Answer: yes — but the test delta dropped from
  S6's +149 to S7's +37. That's the read-shape signature,
  not a quality drop. Surface area + behavioural coverage
  is comprehensive (every endpoint and every aggregation
  path tested); there's just less to test when there are no
  state machines, no atomic-spawn integration, no per-row
  audit-shape fixtures. **First sprint to fall below the
  +60 test floor** of S1..S6 — and that's correct, not
  alarming.
- **B-NNN inline discipline held cleanly for the second
  sprint running.** S6 was the first sprint to honour
  "v1.1 candidate decisions land in the same commit as the
  decision". S7 carried six B-NNN entries into the backlog
  inline (B-055, B-056, B-057, B-058, B-059, B-060). Closing
  audit was 0 entries. T-S5-08-style retroactive sweep: not
  needed.
- **Pure-function pattern extends to read-shape work.**
  `Core/CustomReportRunner.cs` is a 173-line static class
  with a per-entity field allow-list dictionary + JSON
  validators + reflection-based match/project — exactly the
  same shape as `Core/Cpm.cs`, `Core/Evm.cs`,
  `Core/RiskMatrix.cs`, `Core/StakeholderMatrix.cs`,
  `Core/EvaluationMatrix.cs`. Eight Core/ pure-function
  modules across S0..S7 now follow the same shape: no IO,
  no DI, no DB, fully unit-testable. The pattern works for
  arithmetic (CPM, EVM) AND for query interpretation
  (CustomReportRunner). The S6-retro hypothesis that pure-
  function shape generalises was confirmed.
- **Honest proxies beat fake metrics.** KPI cards include
  cards labelled `Cost Spent vs Budget` and
  `Schedule Completion` rather than fake CPI/SPI numbers
  derived from data we don't have (per-line earned-value
  progress). Subtitles call out which cards are proxies.
  This is a **policy preference** to surface in retros: when
  the spec asks for a metric we can't honestly compute,
  ship the closest honest proxy + flag it explicitly + add
  the v1.1 backlog entry — never fake the metric.
- **PDF deferral was the right call.** Option A trade was
  a Ch 24.6 24-hour wait (would have blocked T-S7-03 mid-
  sprint) vs trimmed F.8 bullet (PDF render layer deferred,
  data aggregator delivered). Sprint compressed to one
  session; B-055 carries the PDF library decision +
  reconciliation against canonical PAFM Ch 30 layout when
  it's pasted. Deferral let the sprint close cleanly.

### What did not go well

- **Smoke walk became a mental walk-through rather than a
  literal smoke script run.** Earlier sprints (S3..S6 per
  the conversation summary) had `smoke-sN.ps1` PowerShell
  scripts that exercised the new endpoints against a
  running dev server and a real SQL Server connection.
  S7's "smoke walk" (in T-S7-07) was a tabular cross-check
  of the 808-test surface against the F.8 DoD bullets,
  documented in the sprint log Day 2 entry. **For a read-
  shape sprint with extensive xUnit coverage of the
  service surface, the table-driven mental walk is
  proportionate** — there's no schema-migration smoke or
  cross-process integration to verify that tests don't
  already cover. But this is now a discipline divergence
  from S3..S6 worth recording: from S8+, write an actual
  smoke-sN.ps1 again unless the sprint shape clearly
  doesn't need it.
- **The disabled scheduled remote agent is finally
  worth not mentioning** — per the S6-retro rule "only
  flag in retro if a status change has occurred". Status:
  unchanged. So this is the last retro paragraph
  acknowledging it; the rule itself is now self-enforcing
  and this paragraph also retires.

### What surprised me

- **The CustomReportRunner allow-list discipline made unit
  tests trivial.** Strict per-entity field allow-list at
  write time means the Run path is dead simple — just
  iterate, MatchesFilter, ProjectColumns. The 11 service
  tests passed first try including the
  cross-tenant 404 and the "empty columns = full allow-
  list" semantic. The validation-at-write-time pattern is
  the same one that worked for EvaluationMatrix's weight-
  sum-at-matrix-calc-time decision in S6 — push the
  validation to the right boundary and the run path
  simplifies.
- **Reflection-based query interpretation was 173 lines
  total.** Includes the per-entity allow-list, JSON
  validators, equality interpreter (handling strings,
  ints, decimals, DateTime, Guid, bool, enums as
  string-or-int), and column projector. The expected
  story was "pure-function patterns might not extend to
  query work"; the actual shape was that they extend
  cleanly. The S6-retro extension prediction was right.
- **The PMI 5x5 risk severity bucket boundaries fell out
  cleanly.** High ≥ 15, medium 7-12, low ≤ 6. Score = P×I
  on a 5×5 matrix has unreachable products (e.g. 13, 14
  don't appear) so the boundaries don't need fractional
  thresholds. Buckets line up with standard heat-map
  colour bands. Per-tenant thresholds remain S14 territory
  per the original S2-kickoff Top-3 risk #3 mitigation.

### What I will try differently next sprint

- **Re-introduce `smoke-sN.ps1` as a discipline default
  from S8 onwards.** S7's mental walk-through was
  proportionate but the S5-retro rule was specifically
  about running a smoke script, not just tabulating test
  coverage. From S8: write a proper PowerShell smoke
  script unless the sprint shape clearly needs only the
  test-coverage walk-through (justify in the kickoff if
  skipping). Default = script.
- **Promote B-027 (CI hardening) decision to S8 kickoff.**
  S6 retro flagged "S7 either starts as F.8 Reporting OR as
  B-027 CI hardening; default plan F.8 first". S7 took the
  F.8 path and shipped cleanly. Now is the moment for B-027
  to get an honest re-evaluation: does S8 ship F.9
  Procurement-completion / F.10 Cost-completion / F.X
  next, OR does S8 land as a dedicated CI hardening sprint?
  Test count is 808 — close to the v1.0 pilot ~1000 target;
  CI green-on-PR is the genuine v1.0 deployability gate.
- **Audit the v1.1 backlog at S8 kickoff for fast-followers.**
  The backlog now has **60 entries (B-001..B-060)**. S6
  retro asked for the same audit at S7 kickoff and it was
  partially honoured (kickoff pre-flagged 5 candidates);
  S8 kickoff should do an explicit pruning pass.
- **Consider whether KPI cards / dashboards need a UI
  layer in S8 or wait for S15.** v1.0 pilot can render the
  JSON DTOs in a basic Blazor page; the polished UI lives
  in S15 Mobile. A "minimal Blazor page that renders the
  six dashboards + MPR + KPI cards from the JSON
  endpoints" might be a 4-6h fast-follower in S8 if the
  sprint shape allows.
