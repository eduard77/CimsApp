## Retrospective

### What went well

- **Triple-decision kickoff held cleanly through implementation.**
  All three Option A defaults (GT soft immutability, MOR
  manual flag, Safety Case JSON aggregator) remained correct
  through coding. None required revision. The kickoff
  triple-decision pattern from S8 (one ADR-shaped decision)
  scaled to three kickoff decisions in S10 without trouble.
- **Pattern reuse from S6 / S9 was extreme, again.**
  GatewayPackageWorkflow re-applied CompensationEventWorkflow's
  shape verbatim — same dictionary structure, same role-floor
  enforcement. Per-type sequential numbering (GW1-NNNN /
  GW2-NNNN / GW3-NNNN) re-applied the V1 / TP-NNNN /
  CON-NNNN / CE-NNNN pattern. The MOR + SafetyCaseSummary +
  GT additions reused the same DI shape, the same
  audit-twin pattern, the same cross-tenant 404 test. Most
  of the work was typing.
- **The S8 CI hardening jobs paid off again.** This is the
  second feature sprint validated end-to-end by SQL Server
  smoke + migration round-trip. The big multi-entity
  AddBsa2022GoldenThread migration applied + reverted +
  re-applied without surfacing any broken Down(). Same as
  S9's AddMidpTidp validation — the kickoff Top-3 risk
  ("broken Downs") was falsified again. EF auto-generated
  Down() methods are reliably correct; 2-for-2 sprint
  validation since S8 landed.
- **`// BSA 2022 ref:` doc-comment discipline hit the right
  spot.** Every spec-anchored field on Project, GatewayPackage,
  MandatoryOccurrenceReport, Document carries an inline
  reference to the BSA 2022 section. Pre-pilot legal review
  has a clean target list. Worth recording as a pattern for
  future statutory-domain modules (S11 UK GDPR is the next
  candidate).
- **First sprint where I batched multiple T-SX-NN tasks into
  single commits — and it worked.** T-S10-03..06 shipped as
  one feat + one test commit instead of four feat + four
  test pairs. The four entities share enough infrastructure
  (DbContext, DI, role-matrix table) that splitting them
  forced artificial commit boundaries. Honest finding:
  multi-entity sprints with similar shape can collapse the
  task-per-commit rule when the entities ship together.

### What did not go well

- **Test count delta felt low for the scope.** +22 tests vs
  ~5 new entity types + 11 endpoints + a state machine.
  Comparable sprints (S6 procurement) added 149 tests for a
  similar entity count. Honest reading: the tests cover the
  surface (workflow + audit + cross-tenant) but don't
  exercise per-state edge cases the way S6's
  CompensationEventWorkflow tests did. **Risk:** fewer tests
  on BSA 2022 means less confidence at pre-pilot legal
  review. **Mitigation if it matters at pilot time:** add a
  T-S10-NN-supplemental task in v1.1 that beefs up
  per-Gateway-state edge-case coverage.
- **Audit-event JSON-property-casing inconsistency surfaced
  in T-S10-02.** AuditService uses default JsonSerializer
  (PascalCase) but the rest of the API uses camelCase via
  controller-level options. Created a small surprise when
  writing the audit-shape assertion test. Honest finding:
  this casing inconsistency exists across all AuditService
  detail blobs since S0; it's been fine because most tests
  don't peek into Detail JSON. Worth noting for future
  audit-shape tests; not worth fixing v1.0.

### What surprised me

- **GT soft-immutability scope held to one method.** I'd
  expected the immutability guard to spread across multiple
  service paths (Edit + Transition + SoftDelete + …). In
  practice DocumentsService only exposes Create + Transition
  publicly; the guard sits in TransitionAsync alone. v1.0
  immutability surface is one if-statement. Worth recording.
- **The kickoff scope-cuts on Option A held without
  pressure to revisit.** Halfway through implementation
  there was no urge to add programmatic BSR push or
  canonical Schedule 5 wording. The deferrals genuinely
  sit on the v1.1 side of the line. Calibrates well — I'd
  half-expected one of the three to want more in v1.0.
- **All 21 BSA 2022 tests passed first try.** Including the
  state-machine edge cases (skip-from-Drafting rejected,
  TaskTeamMember role rejected for Submit). Pattern reuse
  + first-try-green is now the routine outcome for
  workflow-shaped sprints.

### What I will try differently next sprint

- **Multi-entity sprints can collapse task-per-commit.**
  S10 Successfully batched T-S10-03..06 as one feat + one
  test commit pair. From S11 onwards: when 3+ entities ship
  together with shared infrastructure, batch them. Single-
  entity tasks keep the per-task commits. Discipline change
  for the kickoff template: estimate as separate tasks but
  commit as a batch when the entity-shape similarity warrants.
- **Statutory-domain modules need test-density bumps.** v1.0
  BSA 2022 ships with +22 tests. For pre-pilot pilot deploy
  to a real customer with HRB exposure, this would benefit
  from an extra 30-50 edge-case tests. Honest carry-forward
  to a v1.1 fast-follower (B-076 candidate if needed).
- **S11 scope decision** (carrying forward):
  - F.11 (post-S8 renumbered = original S10 = "UK GDPR"):
    ROPA, DPIA, SAR intake / fulfilment, breach log with
    72-hour clock, retention schedules.
  - Or: B-NNN backlog audit pass (now at 75 entries).
  - Or: second-pass CI hardening (B-061..B-064).
  Default lean: F.11 module work. UK GDPR is another
  statutory-domain sprint with similar shape to S10 —
  carry the `// GDPR ref:` doc-comment pattern.
