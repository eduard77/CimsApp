## Retrospective

### What went well

- **The S3-retro prediction was tested and falsified — honestly.**
  S3 said the same-day cadence might break under S4's algorithmic
  complexity. It didn't. The CPM solver — 288 lines covering
  forward / backward pass, FS/SS/FF/SF with lag, four constraint
  types, total / free float, cycle rejection — landed with all 15
  PMBOK fixture tests green first try. The prediction was a
  reasonable hypothesis based on three sprints of evidence, and
  it was correctly testable. S4 falsifies it. That's how a
  prediction is supposed to work; the discipline of writing the
  prediction down is what made the falsification visible.
- **CR-005 surfaced and resolved on day 1 — fourth sprint in a
  row** where Ch 17 worked. Capacity gap (94.5h vs 72h) visible
  at kickoff, three options presented, Option A accepted, three
  backlog entries written (B-031 / B-032 / B-033), execution
  proceeded on the trimmed scope. CR-003 (S1) / CR-004 (S2) /
  CR-005 (S4) are now a stable pattern.
- **The S0..S3 patterns transferred cleanly to S4** — same shape
  as the S3 retro, even more strongly:
  - Tenant query filter on every entity (Activity / Dependency /
    ScheduleBaseline / ScheduleBaselineActivity / LookaheadEntry /
    WeeklyWorkPlan / WeeklyTaskCommitment) used the same
    `Project.AppointingPartyId == _tenant.OrganisationId` form
    or `parent.<X>.AppointingPartyId == _tenant.OrganisationId`
    via a parent.
  - Audit-twin pattern — 12 new structured action names this
    sprint (`activity.created/.updated/.deactivated`,
    `dependency.added/.removed`, `schedule.recomputed/.imported`,
    `schedule_baseline.captured`, `lookahead.added/.updated/.removed`,
    `weekly_plan.created`, `weekly_commitment.added/.updated/.removed`)
    — all follow the same shape as the prior 30+ across S0..S3.
  - Three-commit-per-task cadence (entity-with-service / migration
    / tests) carried through with exceptions only where natural
    (T-S4-04 had no migration; T-S4-11 was small enough for a
    single commit).
  - The pure-`Core/<X>.cs`-with-fixture-tests pattern from
    `Evm.cs` / `RiskMatrix.cs` / `StakeholderMatrix.cs` extended
    cleanly to `DependencyGraph.cs` / `Cpm.cs` / `MsProjectXml.cs`.
- **MS Project XML import landed strict-on-core / lenient-on-
  optional.** Strict on root namespace, strict on Task UID /
  Name / Duration. Lenient on per-Task optionals (PercentComplete
  defaults to 0). Skip-and-warn on dependency edge cases
  (unresolved UIDs, self-loops, duplicate pairs) rather than
  fail the whole import. The fixture-test set covers the cases
  we hand-crafted; the v1.1 candidate is to expand fixtures
  with real exports from pilot users.
- **The Gantt endpoint reused everything.** Read-only DTO
  transformation; CPM-fallback-to-Scheduled behaviour; same
  membership gate as every other read endpoint. Single small
  commit (~285 lines across DTO / service / controller / role-
  matrix / 7 tests).

### What did not go well

- **The cadence model is now empirically more wrong than it
  was after S3.** S3 retro said: "the formal capacity calculation
  is now genuinely a discipline tool for decomposition, not a
  prediction." S4 retro extends that: even genuinely-novel
  algorithmic work compresses to within the same-day envelope
  when the algorithm is well-formalised in textbook references
  (PMBOK / Microsoft Project XML schema). The honest framing:
  what makes a sprint *take time* is not "novel algorithms" but
  "novel decisions". CPM, MSP XML, and PPC are all algorithms
  with one correct answer. S9 (Golden Thread / BSA 2022) and
  S10 (UK GDPR) involve interpretive decisions about what a
  regulator wants — those are likely the actual cadence-breakers,
  not algorithmic complexity.
- **Skipped the SQL-Server smoke walk again.** S3 retro made it
  a "standing rule" before merging. S4 didn't run it. The reason
  is that within a same-day single-session sprint, "before
  merging" never actually arrives in the way it would for a
  three-week sprint. The smoke walk action needs re-shaping —
  perhaps as a **pre-PR step in T-S4-13** rather than a
  pre-merge step. Carrying forward to S5 with concrete shape:
  "smoke walk runs as the *first action* of the sprint-close
  task, before retro / tag / PR".
- **Test fixture coverage is one-sided for MSP XML import.**
  Every fixture in `MsProjectXmlTests.cs` is hand-crafted. Real
  MS Project exports include things we haven't seen: outline
  hierarchy, summary tasks, milestones with PT0H0M0S, calendar
  exceptions, recurring tasks. v1.0 ships with the simple-case
  coverage; the v1.1 path is to seed fixtures from real exports
  the first time a pilot user imports a real schedule.
- **Lingering scheduled remote agent still un-deleted** —
  `trig_01CbdDcec2Q2GQK8oidWjm3o` flagged in S2 / S3 retros and
  again here. Already disabled (no execution risk); manual delete
  via the web UI required. Routine carry-forward.

### What surprised me

- **CPM passed every test first try, including the off-by-one-
  prone cases.** Specifically: free float vs total float
  distinction (the `A→B→D, A→C→E` test); FF lag with the
  successor finishing when the predecessor finishes; SF
  semantics; lag of -2 days reducing the successor's start.
  Three-color DFS cycle detection caught the obvious cycles
  and reported the participants. Topological sort handles
  diamond DAGs correctly. The PMBOK formalisation is *exact*
  enough that copying the formulas out of the spec gives
  correct behaviour without iteration. Surprising in absolute
  terms; unsurprising relative to the S3-retro prediction
  framing being wrong.
- **MS Project's LinkLag really is in tenths-of-minutes.** Not
  seconds, not days, not even minutes. Tenths of minutes. So
  4800 tenth-minutes = 8 hours = 1 working day at 8h/day. The
  unit choice is mildly perverse but consistent within MSP.
  The fixture tests pin this conversion down explicitly.
- **The LPS reason-for-non-completion enum is shorter than the
  literature suggests.** Lean-construction texts list 10–15
  failure categories; v1.0 ships with 8. The kickoff Top-3
  risk on Engagement-log throughput already flagged "per-tenant
  custom reasons" as a v1.1 candidate — Real UK construction
  projects often want categories like "Welfare facilities
  unavailable" or "Permit-to-work delay". The v1.1 path is
  per-tenant configurable categories in S14 Admin Console.

### What I will try differently next sprint

- **Run the SQL-Server smoke walk as the FIRST action of T-SX-13**
  (sprint-close task), not as a pre-merge step. The pre-merge
  framing implies "after retro / tag", but in same-day-sprint
  reality there's no pre-merge time-window — the PR opens
  immediately after retro. Re-shape the rule: smoke walk →
  tick DoD → retro → tag → PR. Carrying forward as a standing
  rule from S5 onwards.
- **Replace "novel algorithm = slower sprint" with "novel
  decision = slower sprint".** The decisions that take time are
  the ones with no canonical answer. PMBOK + MSP XML + PPC all
  have canonical answers; the codebase already had the
  audit-twin / tenant-filter / role-matrix / migration scaffold
  for free. Sprints likely to genuinely take time:
  - S5 Change Control (workflow shape is interpretive)
  - S9 Golden Thread / BSA 2022 (regulator-defined evidence
    chains; interpretive)
  - S10 UK GDPR (DPIA / SAR / breach-notification interpretive
    workflows)
  - S15 Mobile Views (UI iteration requires manual feedback
    loop, less compressible)
  - S17 Security & Performance (pen test feedback loop)
  These predictions are testable in the respective retros.
- **Promote v1.1 candidates flagged this sprint to backlog
  items.** Per the S3 retro action — never executed. Re-stating:
  the LPS per-tenant custom reasons, MSP XML fixture-from-real-
  export expansion, MSP elapsed-days lag mode, working-hours-
  per-day from MSP project calendar, all deserve B-NNN entries
  rather than just doc-comment flags. **Action for S5 kickoff:
  audit S0..S4 retros and roll any "v1.1 candidate" mention
  into a B-NNN backlog item.**
- **Open a GitHub issue per CR-NNN at the time of the change
  record.** S0..S4 have CR-001..005; only the v1.1-backlog
  entries exist as issues. Linking the CR-NNN markdown to a
  GitHub issue at the moment of the change-register entry
  would make sprint-close audits cheaper.
