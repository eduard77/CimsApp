## Retrospective

### What went well

- **CR-003 surfaced and resolved on day 1.** The kickoff F.2
  backfill made the capacity gap visible (126h vs 72h formal);
  Option A scope cut was raised, accepted, and recorded on
  the same day rather than absorbed silently. PAFM-SD Ch 17
  worked as designed — same as S0's CR-002 (security review
  hardening) but for capacity rather than scope discovery.
- **The audit-twin pattern generalised cleanly.** Every
  cost-domain mutation lands two audit signals: the
  `AuditInterceptor` per-row Insert/Update/Delete (field-level
  before/after) plus a structured `AuditService.WriteAsync`
  entry for the domain action with semantic detail
  (`{ previous, current }` for budget, `{ rowCount }` for
  imports, `{ cbsLineId, type, amount, reference }` for
  commitments etc.). Eleven new audit actions across S1
  follow the same shape: `cbs.imported`, `cbs.line_budget_set`,
  `commitment.created`, `cost_period.opened` / `.closed` /
  `.baseline_set`, `actual_cost.recorded`,
  `variation.raised` / `.approved` / `.rejected`,
  `payment_certificate.draft_created` / `.draft_updated` /
  `.issued`. No ADR was needed — the pattern documented
  itself in the service layer.
- **One-aggregate-per-service held.** CostService grew to four
  concepts (CBS, commitments, periods, actuals, plus the
  cashflow read added at T-S1-11) but variations and payment
  certificates each got their own service. Rule of thumb:
  if the aggregate has its own state machine, it gets its
  own service.
- **ADR-before-implementation worked.** ADR-0013 (NEC4 default
  contract convention) was written before the T-S1-09 entity
  / service so the decision drove the calculation shape rather
  than retrofitting one. Same pattern as S0's ADR-0010 / 0011 /
  0012; keep doing this when a non-trivial design decision
  appears.
- **PMBOK fixture tests for EVM math.** `EvmTests` exercises
  the canonical PMBOK over-budget / behind-schedule example
  and asserts the textbook property `TcpiToEac == CPI` when
  EAC is the typical formula. These textbook properties are
  cheap regression guards that catch sign errors and divisor
  swaps that decimal-comparison tests would miss.
- **The handoff `.txt` pattern is real now.**
  `s1-handoff-laptop.txt` brought a fresh laptop checkout to
  parity in one structured pass (fetch, install dotnet-ef,
  run all migrations, build, test, document the runtime
  quirk). The pattern earned project memory
  `project_dev_machines.md` for future sessions.

### What did not go well

- **Three deferred features all trace to a single missing data
  primitive.** T-S1-07 EVM service-integration, T-S1-09
  payment-cert valuation, and T-S1-11 per-CBS-line cashflow
  all manually-enter the same data: a per-line scheduled-spend
  curve plus a percent-complete progress signal. These three
  deferrals were taken independently across three different
  tasks; only late did the common cause become obvious. A
  single ADR-or-backlog-item recording "v1.0 has no per-line
  schedule + progress" as the unblocker for all three would
  have been clearer than three independent "v1.1 candidate"
  notes scattered across sprint-log entries. Action: add a
  consolidated v1.1 backlog item (see "Notes for next sprint").
- **The original 126h estimate was off by ~75% against
  capacity.** CR-003 caught the gap on day 1 and trimmed
  scope, which is the right process recovery. But the kickoff
  estimate itself should have been sanity-checked against
  capacity *during* the kickoff F.2 backfill rather than
  surfacing as a "drift risk noticed" in the same commit.
  Action: kickoff template could include a capacity
  cross-check step before the task table is finalised.
- **Laptop dev environment had a runtime mismatch.** The
  installed .NET 8 patch on the laptop is 8.0.11; the
  desktop's tooling reference
  (`Microsoft.EntityFrameworkCore.Tools 10.0.6`) drags the
  requested framework version up to 8.0.26. The
  `DOTNET_ROLL_FORWARD=LatestMajor` workaround runs both
  `dotnet ef` and `dotnet test` on the .NET 10 runtime —
  tests pass identically (130 / 130 across the sprint), but
  the laptop is exercising a .NET 10 runtime for `net8.0`
  -targeted code, not what CI / desktop runs. Captured in
  project memory `project_laptop_runtime.md`. Action:
  install the latest .NET 8 runtime on the laptop before
  S2 starts; the workaround is acceptable but technically
  off-spec.

### What surprised me

- **Velocity again.** The CR-003-trimmed 96h sprint completed
  in roughly 2 working days. As with S0, this is heavy
  AI-paired work on substantially specified ground. The
  numbers make planning hard: the next sprint's capacity
  estimate will once more be guesswork against this pace.
- **The audit-twin pattern became reflexive.** By T-S1-04
  (Budget set), writing the structured AuditService entry
  alongside the interceptor's per-row audit was just default
  reflex. By T-S1-09 (payment certs) the question wasn't
  "should there be a structured audit?" but "what fields
  belong in the detail JSON?" — useful muscle memory for
  the rest of v1.0.
- **The cumulative-net cumulative-chain math was load-bearing.**
  T-S1-09 has a single test that runs three issued payment
  certificates Apr / May / Jun and asserts the conservation
  invariant Σ AmountDue = latest CumulativeNet. That single
  test was the load-bearing one during development — an
  earlier draft of the chain math summed prior AmountDues
  rather than reading the latest prior CumulativeNet, and
  the conservation check was what caught the drift. PMBOK /
  NEC4 are unforgiving about this; the conservation check
  is now the regression guard.

### What I will try differently next sprint

- **Roll up service-integration deferrals into a single
  backlog item early.** When T-S1-07 noted "service
  integration deferred, needs schedule + progress", that
  should have been a v1.1 backlog item written then, with
  T-S1-09 valuation and T-S1-11 per-line cashflow added to
  it as they hit the same blocker. Doing this in S2 will
  reduce noise in per-task sprint logs and surface
  cross-cutting v1.1 work earlier.
- **Capacity-check the kickoff task list explicitly.** Add a
  step to the sprint kickoff: after the task table is
  populated, sum the estimates × 1.5 PAFM E margin and
  compare to the formal capacity. If the gap is > 10%, a
  CR is mandatory before any task is started. The spirit
  was honoured this sprint; the letter belongs in the
  template.
- **For NEC4-shaped vs JCT-shaped decisions, write the ADR
  first.** ADR-0013 was written before T-S1-09 and made the
  implementation tractable — calculation rules followed
  directly. Apply the same when S5 Change Control needs
  Construction Act semantics (B-014).
- **Verify the laptop's runtime patch matches the desktop's
  tooling target.** A 5-minute install would have removed
  the `DOTNET_ROLL_FORWARD` workaround entirely. Action item
  for before S2 kickoff.
