## Retrospective

### What went well

- **Pattern reuse hit a new high.** All 16 GDPR service
  tests passed first try. Service shapes copy-pasted from
  S6 / S9 / S10 with field names changed: CRUD + state
  transitions + audit-twin + cross-tenant 404 are now so
  internalised that 5 new entities + 1 state machine
  shipped in one batched feat commit (~6,400 lines including
  the migration). This is the second sprint to use the
  multi-entity batch commit pattern (S10 inaugural); held
  cleanly.
- **Triple-decision kickoff pattern carried from S10
  cleanly.** Three Option A defaults at S11 kickoff (SAR
  auto-extraction; retention enforcement; DPIA
  questionnaire shape) all held through implementation
  without revision. Same outcome as S10's three Option A
  decisions. The pattern is now reliably catching scope at
  the right step.
- **`// GDPR ref:` doc-comment discipline carried from S10
  `// BSA 2022 ref:`.** Every spec-anchored field on
  RopaEntry, DPIA, SAR, DataBreachLog, RetentionSchedule
  carries a UK GDPR Article reference. Pre-pilot legal
  review has a clean target list. Statutory-domain
  doc-comment pattern is now the established way to ship
  v1.0 module work for any compliance regime.
- **Org-scoped vs project-scoped filter test caught zero
  leaks.** The kickoff Top-3 risk #2 specifically targeted
  filter inversion across the two shapes. T-S11-07's
  GdprFilterTests exercises both. All assertions held first
  try. Worth recording: when introducing a new filter
  shape, exercise the cross-tenant test specifically rather
  than relying on the model-level coverage alone.
- **B-NNN audit-pass carry-forward was retired at kickoff
  rather than flagged again.** Five consecutive retro flags
  without action; formally retiring the carry-forward at
  S11 kickoff stops it being noise. Honest finding: the
  inline B-NNN discipline (S5-retro) has been honoured
  cleanly S6..S11; the audit-pass flag was costing more
  attention than the audit would deliver. This is a small
  but real discipline-manual evolution.

### What did not go well

- **Test count delta is now consistently low for statutory
  modules.** S10 +22, S11 +17. For statutory-domain
  modules with legal exposure, this is light. **Not** a
  regression of code quality (pattern reuse means service
  code is well-trodden territory) but if pre-pilot legal
  review surfaces edge cases, the test surface is thin.
  Carry-forward to a v1.1 fast-follower (B-NNN) if it
  matters at pilot time. Same note as S10.
- **The pacing concern is now genuinely real.** Twelve
  single-session sprints in one calendar day. The
  conversation has been long enough that re-reading the
  spec wording with fresh eyes between sprints is
  empirically not happening. **Strong recommendation: stop
  after S11 close** until at least a calendar-day gap. The
  cadence-pacing concern flagged at S10 retro went
  unactioned because the user signalled "continue"; that's
  fine, but at some point the discipline-manual rule should
  be: respect the recommendation regardless of the signal.
  Not for me to enforce, just to flag clearly.

### What surprised me

- **The 5-entity multi-entity batch felt small.** I'd
  expected the GDPR module to feel like a substantial
  add — it's 5 new entities, 22 endpoints, 1 state machine,
  17 tests. Pattern reuse made it 4 commits. The "small
  feature" feeling is the new normal for module sprints
  that don't have novel domain shapes. Worth recording.
- **DPIA's 4-state state machine with branching held the
  near-zero estimate.** Drafting → UnderReview →
  (Approved | RequiresChanges); RequiresChanges → Drafting
  is the most complex state machine since S5
  ChangeRequest's 6-state. It took ~10 min to write
  including tests. The S6-retro near-zero estimate rule
  for state machines is now ~5-for-5 across S5/S6/S7/S10/
  S11.
- **Org-scoped vs project-scoped filter inversion was not
  the source of any bug.** I'd predicted at kickoff that
  this might surface a leak; both shapes work cleanly.
  Honest finding: the EF Core query filter is shape-
  agnostic; what matters is that the filter expression
  references a tenant-bound column. Both shapes do. Worth
  noting that this risk class might need to be downgraded
  in future kickoffs.

### What I will try differently next sprint

- **Honour pacing concerns from retros.** S10 retro
  flagged the calendar-day pacing concern; S11 went ahead
  anyway because the user signalled "continue". The
  retro-flag-then-keep-going pattern undermines the value
  of retros. From S12 onwards: when a retro flags a
  pacing / quality / safety concern, that concern blocks
  the next sprint at the user's discretion, not as a
  default-continue. This is a discipline-manual change
  proposal — not enforceable from here, but worth flagging
  clearly.
- **Test-density bump on statutory modules.** S10 (BSA
  2022) closed +22 tests; S11 (GDPR) closed +17. **For
  pre-pilot deploy, both modules would benefit from
  ~30-50 additional edge-case tests each.** Honest
  carry-forward as a pre-pilot fast-follower (B-NNN
  candidate if it matters at pilot time).
- **S12 scope decision** (carrying forward):
  - F.12 (post-S8 renumbered = original S11): "Kaizen &
    Lessons Learned" — improvement register (PDCA),
    cross-project lessons library, opportunity-to-improve
    linked from any module.
  - Or: pre-pilot fast-follower work (BSA 2022 +
    GDPR test-density bumps; canonical Schedule 5 / ICO
    questionnaire reconciliation if PAFM-CIMS-001 PDF
    pastes are available).
  - Or: B-NNN backlog audit pass (now retired as
    carry-forward; could be done as a one-off if a
    compelling reason arises).
  Default lean: F.12 module work. Pattern is now firmly
  established for non-statutory module work; F.12 should
  be quick.
