## Retrospective

### What went well

- **Kickoff with capacity headroom.** First sprint since S0 to
  start with the formal estimate (45h × 1.5 vs 72h capacity)
  sitting comfortably below capacity — no CR-005 needed. Validates
  the kickoff template's discipline: visibility on capacity drives
  better scoping decisions, even when (especially when) no cut is
  needed.
- **The S0 / S1 / S2 patterns transferred cleanly to S3.** The
  Mendelow Power/Interest grid mirrored the S2 P×I risk matrix
  structurally — `Core/StakeholderMatrix.cs` is a near-twin of
  `Core/RiskMatrix.cs`, same 25-cell row-major build, same
  `[id]` per cell. The audit-twin + role-matrix update protocol
  applied without modification across 11 new endpoints (4 on
  Stakeholders, 2 on Engagements, 4 on Communications, 1
  Stakeholder matrix). Three-commit-per-task cadence (entity-with-
  service / migration / tests) carried through.
- **Engagement-plan-as-two-fields was the right call.** Spent 5
  minutes at kickoff considering an explicit `EngagementPlan`
  entity with action-list / cadence / metrics; chose
  `Stakeholder.EngagementApproach` + `EngagementNotes` instead.
  v1.0 pilots can populate the engagement plan without a join
  table, and v1.1 has a clean upgrade path if real workflows
  demand structured plans. The spec ambiguity ("engagement plan
  per stakeholder") was resolved with the simplest thing that
  could satisfy F.4.
- **Resisted three pattern-deviation temptations mid-sprint.**
  (1) Pre-building S14 Admin Console threshold-override scaffold
  while writing StakeholderMatrix doc-comments. (2) Adding a
  `?since=` query parameter to EngagementLog listing. (3) Linking
  CommunicationItem.Audience to Stakeholder rows. All three would
  have been small extra steps; all three are documented v1.1
  candidates. Same discipline lesson as the S2 retro — when
  deferral feels easy at the moment of decision, the kickoff
  Top-3 risks list is what protects the boundary.
- **Owner-must-be-project-member validation on CommunicationItem
  was a real defensive add.** Caught while writing the test
  fixture — a CommunicationItem with an outsider Owner is
  semantically broken (the Owner can't read the communication
  matrix they own). The validation is two lines (`AnyAsync` on
  ProjectMember); the test fixture was the source of insight.

### What did not go well

- **Sprint cadence is now stably anomalous, not anomalous.** Four
  sprints of evidence: same-day completion regardless of formal
  estimate (S0: 49h, S1: 96h post-cut, S2: 75h post-cut, S3: 45h).
  The S2 retro said "track velocity differently — perhaps modules
  closed per session"; S3 confirms that's the right metric. The
  formal capacity calculation now genuinely is (per the S2 retro
  action) a discipline-tool for decomposition, not a prediction.
  The retro action carries forward: S4 Schedule & Programme is
  the test — CPM algorithms have algorithmic complexity that the
  existing patterns *don't* help with, so velocity should drop.
  If S4 still completes in one session, the model is broken
  regardless of the sprint subject.
- **No SQL-Server smoke walk this sprint.** S2 retro flagged this
  as a queue-for-S3 action ("Run security-review at midpoint, not
  just close"); didn't happen because there was no midpoint.
  Carrying forward: same shape as the S1 SQL-Server smoke walk
  that found four production bugs, but for S3 the surface area
  (Stakeholder + EngagementLog + CommunicationItem with the
  membership-validation cross-table FK on CommunicationItem)
  isn't trivial. Action: run the smoke walk before merging the
  S3 PR. Bootstrap → register → login → create project → create
  stakeholder → record engagement → create comm-item → list /
  matrix.
- **Scheduled remote agent still un-deleted.**
  `trig_01CbdDcec2Q2GQK8oidWjm3o` from S0; flagged in the S2
  retro and again this sprint. Already disabled (so no execution
  risk) but still cluttering the routines list. Manual delete
  via https://claude.ai/code/routines required (API doesn't
  support delete from the agent side).

### What surprised me

- **The Communications matrix entity was simpler than expected.**
  Pre-decomposition I'd assumed `CommunicationItem` would need
  a join table to link Audience to Stakeholder rows. The S3
  kickoff Decisions section moved that to v1.1, and the
  free-text Audience field landed cleanly with no real loss of
  v1.0 functionality — every realistic communication matrix is
  populated by a PM typing roles or stakeholder names anyway.
  v1.1 can add the structured link without breaking anything.
- **The `EngagementLog` 200-row cap was easier to defend than
  expected.** No real-world stakeholder has 200 logged
  interactions in a single project lifecycle; if any does, it's
  a workflow signal worth investigating before adding pagination.
  v1.1 candidate kept honest by the kickoff Top-3 risks entry.
- **`StakeholderMatrix.cs` came in at 60 lines, vs `RiskMatrix.cs`
  at 80.** The S2 RiskMatrix had a separate `RiskScore` static
  helper and a category-scoring branch; the S3 StakeholderMatrix
  doesn't need either (Power/Interest are always 1..5; no
  category dimension on Stakeholder). The pattern compresses
  when the domain has fewer dimensions — note for S4 Schedule
  & Programme where the dimensions multiply (activity / float /
  resource / cost), the pattern likely won't transfer as cleanly.

### What I will try differently next sprint

- **Run the SQL-Server smoke walk between sprint close and PR.**
  Carrying the S2 action forward as a now-mandatory step. The
  per-sprint discipline to add: don't open the PR until the
  smoke walk has been run against real SQL Server. Re-use the
  S1-style script (bootstrap-org → register → login → exercise
  the new entities → confirm audit-twin landing) — adapt for the
  S3 surface area.
- **Pre-flight S4 with an Explore agent before kickoff.** Same
  S2-retro action that didn't get used in S3 because S3 was
  pattern-similar to S2. S4 Schedule & Programme has CPM
  algorithms that are genuinely novel relative to what the
  codebase has shipped. An Explore over the F.5 spec (corrected:
  F.5 in the discipline manual is Schedule per Appendix F's
  S-series) + the current scheduling-adjacent code (cost-period
  date ranges, EVM data-date logic, cashflow period overlap)
  before the kickoff would catch capacity / dependency risks at
  the right step.
- **Stop sandbagging the deferral lists.** The S0 / S1 / S2 / S3
  pattern: kickoff lists three v1.1 candidates per Top-3 risks;
  most never get formally tracked beyond the kickoff doc.
  Action: at S4 kickoff, add explicit B-NNN backlog issues for
  any v1.1 candidate that touches an existing v1.0 endpoint
  (e.g. EngagementLog pagination would be a backlog issue;
  org-level templates is enough of a forward-design step that
  it deserves a backlog tag too). The kickoff doc is fine for
  decision-records; the backlog is what carries items into v1.1
  planning.
