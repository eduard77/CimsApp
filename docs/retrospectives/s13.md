## Retrospective

### What went well

- **Option A scope cut held cleanly.** No mid-sprint pressure
  to revisit, no scope creep, no pilot-impact regret. The
  honest framing ("Genera sync deferred to v1.1 / B-086..089
  pending Ch 47 paste") works as a v1.0 delivery shape; the
  alternative (stub schemas) would have been worse.
- **State machine pattern reuse is now mechanical.** 7th
  state machine in CIMS following the established shape;
  ~15 lines + 1h fixture. S6-retro near-zero estimate
  empirically 7-for-7.
- **B-NNN unblock conditions are a cleaner shape than
  before.** B-086 says "Ch 47 paste + Genera API spec
  confirmation"; B-087 says "Ch 47 webhook event schema +
  delivery semantics decision"; B-088 says "Genera IdP
  decision". Each unblock condition is concrete and
  individually testable. Worth carrying forward to future
  deferral entries: state the unblock condition, not just
  the deferral reason.
- **Calendar-day gap from S12 close honoured (partially).**
  S12 closed 2026-05-02; S13 kickoff 2026-05-03. The
  pacing-concern recommendation from S11/S12 retros is now
  empirically followed — at least for one calendar day.

### What did not go well

- **+8 test count is the lowest of any feature sprint** —
  by design, but worth recording. Most of the would-be test
  surface (REST API contract tests, webhook event-schema
  tests, identity-sync federation, bidirectional conflict
  resolution) defers to v1.1 alongside B-086..089. v1.0
  pilot users get a working manual workflow with adequate
  edge-case coverage for that workflow shape. Pre-pilot
  legal review concern doesn't apply (F.13 isn't statutory).

### What surprised me

- **The kickoff Option A discussion was easier than
  expected.** Stating the deferral cleanly + listing the
  four B-NNN unblock conditions felt straightforward. The
  framing pattern ("v1.0 ships data shape; integration
  deferred to v1.1 / B-XXX with [unblock condition]") is
  general-purpose — applies to any module where one or more
  bullets depends on external paste-on-request material.
- **No need for a triple-decision kickoff.** S10/S11 had
  three Option A defaults each; S12 carried the pattern;
  S13 is single-decision (Option A on the whole module).
  When the deferral is across the entire module rather than
  per-bullet, the kickoff is structurally simpler.

### What I will try differently next sprint

- **Carry the "unblock-condition per B-NNN" pattern.** S13
  introduced it cleanly; future sprints with deferrals should
  state the unblock condition explicitly (not just "will ship
  in v1.1"). It changes a passive carry-forward into an
  actionable item with a clear trigger.
- **S14 scope decision.** Default lean per the post-S8
  renumbered roadmap is F.14 Notifications & Alerts (in-app
  via SignalR, email, SMS optional, threshold-based alerts).
  Non-statutory, infrastructure-shaped — different domain
  from the recent module sprints. Likely needs a single
  ADR-shaped decision on transport / delivery semantics.
