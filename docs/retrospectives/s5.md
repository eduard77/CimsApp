## Retrospective

### What went well

- **The S4-retro prediction was tested and falsified again, more
  cleanly this time.** S4 said S5 wouldn't close same-day because
  the workflow shape is "interpretive". The state machine landed
  in 280 lines (Core + tests) with zero design back-and-forth.
  Honest framing: the v1.1-deferred items (B-035..B-038) are
  exactly the points where genuine interpretation *would* have
  been needed (per-tenant delegation? BSA inference rules?
  expedited-path? structural impact links?) — and those items
  were correctly identified at kickoff and parked in the backlog.
  v1.0 ships the canonical-PMBOK / canonical-NEC4 core; v1.1
  ships the per-tenant / per-customer interpretation layer.
- **Two sprints in a row with no CR raised** (S3 + S5; S4 had
  CR-005 day-1). The kickoff capacity-check + scope-discipline
  conversation is now stably catching capacity gaps at the right
  moment. S3 (45h estimate, 27h headroom) and S5 (37.5h estimate,
  34.5h headroom) both fit comfortably; S0 / S1 / S2 / S4 all
  needed CRs at kickoff. The pattern is stabilising: **smaller
  F.X DoDs (4–6 bullets) tend to fit; larger F.X DoDs (7+
  bullets) tend to need CRs.**
- **The Approve→Variation spawn integrated cleanly across two
  modules** (S5 ChangeRequest ↔ S1 Variation) without any
  refactoring of S1 code. The 5 cross-service integration tests
  in T-S5-06 cover: forward spawn carries impacts, spawned
  Variation drives independently through S1 workflow, FK
  survives the Variation's own state transitions, no spawn when
  flag false, multiple CRs yield distinct VAR-NNNN. The atomic
  side-effect-inside-the-transition pattern is now a
  established shape — likely re-used in S6 Procurement (Approve
  Tender → spawn Contract).
- **The v1.1 backlog audit (T-S5-08) closed a real discipline
  gap.** 14 new entries (B-034..B-047) accumulated across
  S3..S5 doc-comments + retro mentions; rolling them up made
  the v1.1 surface actually visible. Plus B-027 (CI smoke test)
  promoted from open-ended deferral to a pre-pilot dedicated-
  infrastructure-sprint target — the carry-forward had been
  flagged in 5 retros without any concrete commitment until
  now.

### What did not go well

- **Smoke walk skipped *again* — same failure as S4.** S4-retro
  re-shaped the rule to "smoke walk = FIRST action of T-SX-13
  sprint-close". S5 close hasn't run a smoke walk yet at the
  point of writing this retro (the retro is the first close-out
  doc; smoke would happen before the PR step). Action: run
  the smoke walk *before* the docs commit, not after. This
  retro is being written **before** the smoke walk to surface
  the violation, but the smoke walk is the next concrete step
  before any push / PR.
- **The "first calendar-day inflection" is a small but real
  data point.** S5 is the first sprint where the work + close
  span 2 calendar days. The S4-retro framing was about *novel
  decision = slower sprint*; the actual rest-pattern inflection
  is unrelated to the work shape — the user took overnight
  rest between yesterday's code work and today's close. Honest
  read: the technical-work cadence model still holds (~1
  session per sprint of substantive code); the close-out task
  is small enough that splitting it across the day boundary is
  fine — but the "same-day kickoff to close" framing in
  retros 0..4 needs adjusting. Going forward, "same-session
  substantive code" is the more accurate framing than
  "same-day from kickoff to close".
- **Lingering scheduled remote agent still un-deleted** (5th
  retro carry-forward) — `trig_01CbdDcec2Q2GQK8oidWjm3o`. No
  execution risk; manual delete via the web UI required. Roll
  forward.

### What surprised me

- **The state machine was 280 lines including tests.** I'd
  budgeted 3h for T-S5-03 expecting 2h on the state machine + 1h
  on tests. Actual was probably 30-45 minutes total. The
  CdeStateMachine pattern (S0) gave a complete template; the
  only S5-specific code is the transition dictionaries + the
  `IsTerminal` helper. Test fixtures were rote enumeration of
  the transition matrix.
- **The Approve(CreateVariation=true) pattern is more reusable
  than expected.** I wrote it for the F.6 fifth bullet specifically,
  but the same shape applies anywhere a workflow approval needs
  to atomically spawn a downstream artefact: S6 Procurement's
  "Approve Tender → spawn Contract"; S2 Risk's "Mitigate Risk →
  spawn Action Plan"; S9 Golden Thread's "Sign-off → spawn
  evidence record". Likely worth extracting as a `Core/<X>.cs`
  helper at S6 if a second concrete instance lands.
- **B-036 / B-037 / B-038 weren't kickoff-day decisions in the
  pure sense — they were "the moment we hit the question, the
  answer was obviously v1.1".** The kickoff doc made these
  decisions look pre-planned; in reality they were live
  judgement calls during the F.6 doc-comment writing in T-S5-02.
  The discipline of writing the doc-comment with the v1.1
  flag *first* (before the code), then back-propagating it to
  the kickoff doc, made the "no CR needed" claim honest. Worth
  generalising as a sprint-execution rule: "every v1.1
  candidate decision lands as a doc-comment + kickoff-doc
  update + backlog entry within the same commit; never
  discover them in the retro."

### What I will try differently next sprint

- **Run the SQL-Server smoke walk *before* writing the retro,
  not after.** The S4-retro re-shaped the rule. S5-retro
  refines it: smoke walk → tick DoD → write session entry →
  write retro → tag → PR. This retro itself violates that
  ordering (it's being written before the smoke walk); the
  intent for S6 onwards is to run smoke first as a hard
  T-SX-13 step.
- **Land v1.1 candidate backlog entries inline at decision
  time, not as a closing audit.** Per the T-S5-08 audit
  observation: 14 entries accumulated across S3..S5 doc-
  comments because the discipline was "park as v1.1 candidate
  in source comment, deal with it later." S6 onwards: every
  v1.1 candidate decision adds a B-NNN entry to
  `v1.1-backlog.md` in the same commit as the decision. The
  audit task disappears.
- **Reframe the cadence model again.** S0..S5 evidence: the
  same-session compression holds when the spec is well-formalised
  (PMBOK / NEC4 / ISO 19650 / BSA 2022 high-level / MS Project
  XML schema / EVM formulas / CPM textbook). The actual
  cadence-breakers are likely:
  - **Genuine UI iteration** (S15 Mobile Views) — manual
    feedback loops require the user to drive a browser, can't
    compress.
  - **Cross-cutting infrastructure decisions** (B-027 CI
    hardening sprint; S17 Security & Performance pen test
    feedback loop) — runner / Docker / cert / load / vendor
    decisions involve choices outside the codebase.
  - **Customer-onboarding work** (S18/S19 UAT, training,
    documentation) — the loop runs at customer pace.
  These remain the explicit predictions for which sprints
  should genuinely take time.
- **Promote "same-session substantive code" as the canonical
  framing.** Stop describing sprints as "same-day kickoff to
  close" — that's been false in spirit since S5 split across
  2 calendar days. The honest framing: each sprint's
  substantive code work is 1 session; close-out admin can
  fall on a different day without it being a cadence break.
