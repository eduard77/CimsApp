## Decision required at kickoff: PDF rendering for the MPR

F.8 second bullet says: *"Monthly Project Report generation
(PDF) per PAFM Ch 30."* This requires a PDF library NuGet
package — a Ch 24.6 ADR-shaped decision with a 24-hour wait
between adding the package and exercising it.

**Three options:**

**Option A (recommended, default for this kickoff) — Defer
PDF rendering to v1.1.** Ship the MPR data aggregator as a
JSON DTO that the UI / future PDF rendering layer consumes.
Adds backlog entry **B-055 — MPR PDF generation per PAFM
Ch 30** carrying the library-choice decision (QuestPDF,
PdfSharpCore, HTML→PDF, etc) and the PAFM Ch 30 layout
detail (which is in the PDF spec, paste-on-request per the
`reference_pafm_spec.md` authorisation).
Trimmed F.8 fourth bullet still ticked: "MPR generation"
delivered as data aggregation; PDF render deferred. Honest
framing: the MPR data is the hard part; the PDF templating
is the easy part once the library is chosen.

**Option B — Add QuestPDF NuGet via ADR-0015 + 24h wait.**
ADR-0015 records the library choice + community-licence
applicability (Genera Systems is under-1M-revenue per the
QuestPDF Community License at time of writing — verify at
ADR-write time). 24-hour wait blocks T-S7-03 mid-sprint;
T-S7-04 / T-S7-05 can proceed in parallel during the wait.

**Option C — Ask Eduard to paste PAFM-CIMS-001 Ch 30** so we
know the canonical MPR structure before deciding library.
Pure-information option; doesn't move code.

Default: Option A. T-S7-03 ships a JSON-only MPR data
aggregator; PDF render → B-055.

## Top 3 risks (PAFM-SD Appendix E)

1. **PAFM Ch 30 MPR layout is unknown at kickoff.** The PDF
   spec (PAFM-CIMS-001) Appendix is paste-on-request per
   `reference_pafm_spec.md`. T-S7-03 ships a v1.0 best-effort
   MPR structure inferred from PMBOK / NEC4 standard reporting
   templates. **Mitigation:** mark every MPR section with a
   `// PAFM Ch 30 reference: best-effort inference v1.0`
   comment so the v1.1 PDF-rendering work (B-055) can reconcile
   against the canonical Ch 30 paste at that time.
2. **Cross-module aggregation N+1 queries.** Each dashboard
   touches 4-8 modules' DbSets. Naive `await db.Foo.Where(...)
   .ToListAsync()` per module per request can produce
   >10 queries per dashboard load. **Mitigation:** v1.0 ships
   the simple per-module queries inline; if profiling shows
   real latency at pilot, B-NNN candidate is a single combined
   query per dashboard. Keep it simple first.
3. **`CustomReportDefinition` filter / columns persistence
   shape.** Storing `FilterJson` and `ColumnsJson` as serialised
   JSON in DB columns is the simple path; the structured-filter
   alternative (per-criterion table with type-safe operators)
   is over-engineered for v1.0. **Mitigation:** v1.0 stores
   JSON; queries deserialise + compose at run time. Migration
   to structured filters is a v1.1 candidate if a real
   customer hits the typed-query / index-friendly need.
