## Reference-data reconciliation decision

Two ISO 19650 reference-data tables exist on master:
- `CimsApp/Core/Iso19650Codes.cs` (S0-era, 152 lines, used by
  app at runtime). 31 TypeCodes, full SuitabilityCodes incl.
  A1–A5/AB/B1–B3, RoleCodes, RevisionCodes.
- `CimsApp/Services/Iso19650/Iso19650ReferenceData.cs` (parked,
  89 lines, used only by the parked validator). 7 TypeCodes,
  S1–S7 only, RoleCodes, **plus** Uniclass + IFC mappings.

**Decision (kickoff): Core/Iso19650Codes.cs is canonical.** It
has the broader standard-compliant table set; the parked
validator's narrower table was illustrative. T-S9-02 merges
Uniclass + IFC mappings INTO Core (the parts Core lacks),
deletes the parked `Iso19650ReferenceData.cs`, points the
validator at Core. Saves 89 lines of duplicate code; keeps the
broader real-world coverage.

One subtle gotcha worth flagging: the parked validator uses
`MD` for model whereas Core uses `M3`/`M2` per ISO 19650-2
Table A.1 (3D model / 2D model). Existing
`Iso19650FilenameValidatorTests` use `MD` in test strings;
those test fixtures need updating to `M3`/`M2` after the
reconciliation. T-S9-02 owns that fixture rewrite.

## Validator integration policy decision (T-S9-03 entry point)

Activating the validator on `DocumentsService.CreateAsync`
could break existing flows if any currently-permitted
document name fails the strict ISO 19650 ruleset.
**Decision (kickoff): strict-on-new-documents only, no
retroactive validation of existing Documents.** New documents
created after the S9 merge are validated; pre-existing rows
are left alone. Per-tenant relax-to-warning toggle is a v1.1
candidate (B-065).

The validator also has 10 distinct check categories
(check 1 = pattern, check 7 = Uniclass mandatory for object
class, check 9 = IFC schema, check 10 = cross-reference
integrity). v1.0 wires checks 1–6 (pattern / code-set /
suitability / revision); checks 7–10 (Uniclass / IFC / cross-
reference) are deferred — they need an authoritative NBS feed
which is itself a v1.1 / B-068 item.

## Top 3 risks (PAFM-SD Appendix E)

1. **Reference-data merge regression.** Picking Core/
   Iso19650Codes as canonical means re-pointing the parked
   validator + updating its test fixtures (`MD` → `M3`/`M2`).
   Risk that some existing test asserts an `MD`-flavoured
   string that won't match `M3` or `M2`. **Mitigation:** run
   `dotnet test --filter Iso19650FilenameValidatorTests` after
   each commit in T-S9-02; the breakage will be loud.
2. **Validator integration breaks existing Document creation.**
   The validator was off-limits since session 3; production
   Documents may exist with names that don't validate per the
   strict ISO 19650 ruleset. **Mitigation:** strict-on-new-only
   policy (T-S9-03 decision); no retroactive validation;
   per-tenant relax-to-warning is B-065.
3. **MIDP / TIDP domain modelling overreach.** ISO 19650-2 §5
   has rich semantics (information requirements per
   AIR / OIR / EIR / PIR / BIM authorisation gates).
   Modelling the whole standard at v1.0 burns the sprint.
   **Mitigation:** v1.0 = "list of planned deliveries with
   dates and owners"; full information-requirements model is
   B-NNN territory if a real customer asks. Document the
   scope-cut in MIDP + TIDP doc-comments at T-S9-05/06.
