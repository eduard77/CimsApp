## Sprint goal

Ship CI green-on-PR as the v1.0 deployability gate. Today
the CI workflow at `.github/workflows/ci.yml` runs build +
test on Linux (ubuntu-latest); that catches compile errors
and unit-test failures but **misses the entire FK-permissive
class of bugs** that bit at PR #41 (AuditLog.UserId
non-nullable + Guid.Empty sentinel = production FK
violation, latent for ~a week under unit-test-only
validation because EF in-memory ignores FK constraints).

S8 closes that gap: a SQL Server-backed smoke job runs the
five B-027 endpoints against a real RDBMS, applies all
migrations, asserts HTTP status + response shape, and blocks
merge on failure.

This is the **first sprint where the cadence question is not
"how fast does it compress" but "does the CI runner shape
itself work end-to-end on the first push"** — a pure
infrastructure shape unlike any of S0..S7.

## Decision required at kickoff: CI runner shape

B-027's deferral text explicitly flagged the runner-shape
choice as Ch 24.6 ADR-shaped: *"Setting up a CI job with a
SQL Server service is non-trivial — plus deciding the CI
runner shape (GitHub Actions Windows for LocalDB? Linux
+ mssql? both?)."*

**Three options:**

**Option A (decided 2026-05-02 at kickoff) — Linux runner +
`mssql/server` Docker service container.** Industry-standard
.NET CI shape; free GitHub Actions Linux minutes; ~30-60s
container boot; easy to parallelise with the existing
build+test job. Trade-off: small connection-string
divergence between dev (LocalDB) and CI (TCP to
`localhost:1433`) — handled via workflow-level env var or
`appsettings.CI.json`. Rationale documented in ADR-0015
(T-S8-02).

**Option B — Windows runner + LocalDB.** Mirrors dev
environment exactly. Windows minutes are 2× Linux on private
repos and runners are slower to provision. Rejected at
kickoff.

**Option C — Both Linux + Windows matrix.** Twice the CI
cost + twice the maintenance with marginal value over
Option A. Rejected at kickoff.

Decided: **Option A.** ADR-0015 captures the call.

## Top 3 risks (PAFM-SD Appendix E)

1. **`mssql/server` Docker container readiness probe is
   timing-fragile.** Container boots in 30-60s; the CI
   step that runs migrations before the container is
   ready will fail with "connection refused" or "login
   failed". **Mitigation:** explicit health-check loop
   with `sqlcmd` poll-until-ready + a 90s timeout cap
   before failing. The standard idiom in `actions/runner`
   GitHub-Actions docs is `services.<service>.options:
   --health-cmd "..." --health-interval 5s --health-retries 20`.
   Both layers belt-and-braces.
2. **Smoke endpoint state leakage between runs.**
   Five-step smoke is stateful (org → user → token →
   project list → project create). If two PRs run
   concurrently against the same DB the second sees the
   first's data + cross-pollutes. **Mitigation:** each CI
   run gets a fresh `mssql/server` container (service
   containers are per-job-instance in GitHub Actions);
   plus deterministic seeded values (org code, user
   email) include the run ID so collisions are
   impossible even in shared-DB hypotheticals.
3. **`Down()` methods on existing migrations may be
   latent-broken.** v1.0 has 60+ migrations (all S0..S7)
   and none have ever been exercised in reverse. T-S8-04
   round-trip catches that. Likely outcome: 1-3
   migrations have a broken Down (typically a `DROP
   COLUMN` that should have been a `RestoreColumn`-with-
   default). **Mitigation:** if T-S8-04 surfaces broken
   Downs, fix them inline or — if the fix is structural
   — record an ADR-shaped decision as part of T-S8-04
   itself. Worst case, T-S8-04 surfaces 1 hour of
   broken-Down patches.

## Cadence prediction

S6 retro retired the cadence-breaker section "unless it
actually breaks". S8 is the **first sprint where it
plausibly might** — infrastructure work has different
debugging-loop characteristics (CI iteration cycle is
~2-5min per push vs ~3s per local test run), so even
small mistakes have a higher friction cost.

Honest prediction: **substantive work compresses to one
session if the runner-shape Option A turns out to be
straightforward (which it should — `mssql/server` Docker
in GitHub Actions is well-documented). Compresses to two
sessions if the readiness probe / connection-string /
network-namespace stuff needs a debugging round.** Either
is fine.
