## Sprint goal

Deliver the F.11 UK GDPR module: ROPA (Records of Processing
Activities), DPIA workflow, SAR (Subject Access Request)
intake / fulfilment, data breach log with 72-hour clock, and
retention schedules.

UK GDPR + Data Protection Act 2018 introduces statutory
obligations for organisations processing personal data. v1.0
ships the data shapes + workflows; **statutory legal
accuracy review is not in scope** — paste-on-request from
PAFM-CIMS-001 PDF + legal sign-off step before pilot deploy
(same pattern as S10 BSA 2022).

S11 carries the `// GDPR ref: <article>` doc-comment pattern
established in S10 (`// BSA 2022 ref: <section>`) and
re-applies the **triple-decision kickoff pattern** that held
cleanly through S10 implementation.

## Decisions required at kickoff (triple-decision pattern from S10)

**Decision 1 — SAR auto-extraction scope.**

UK GDPR Art. 15 grants data subjects the right to access
their personal data. Two plausible v1.0 interpretations:

- **A (recommended):** v1.0 captures the SAR intake +
  manual-fulfilment workflow. The actual extraction of the
  subject's data across CIMS domains (User account,
  ProjectMembers, AuditLogs, Documents authored, RFIs raised,
  comments, etc.) is a manual step by the IM / DPO. v1.1 /
  B-077 adds per-domain auto-extractors that compose into a
  SAR fulfilment package.
- **B:** Build per-domain extractors now. Massively more
  work; touches every domain entity. Genuinely v1.1
  territory.

**Default at kickoff: Option A.** Same shape as S10's MOR
manual-push decision — v1.0 captures data, v1.1 adds
programmatic integration.

**Decision 2 — Retention enforcement.**

UK GDPR Art. 5(1)(e) requires personal data to be kept no
longer than necessary. Two interpretations:

- **A (recommended):** v1.0 stores retention schedules per
  data category as a policy document. **No automatic
  deletion** — schedules are reference data. v1.1 / B-078
  wires retention-schedule expiry to per-domain deletion
  triggers (User soft-delete, Project archival, Document
  voiding, etc.).
- **B:** Auto-enforce now. Touches every domain — the
  cross-domain deletion ordering alone needs an ADR.

**Default at kickoff: Option A.** Honest v1.0 — a stored
retention policy is itself a GDPR compliance artefact even
without auto-enforcement.

**Decision 3 — DPIA questionnaire shape.**

UK GDPR Art. 35 requires DPIAs for high-risk processing.
The ICO publishes a canonical questionnaire template.

- **A (recommended):** v1.0 ships the DPIA workflow (state
  machine: Drafting → UnderReview → Approved |
  RequiresChanges → back to Drafting) with a free-text
  Description + structured Risk fields. ICO canonical
  questionnaire schema → v1.1 / B-079 (legal review +
  schema reconciliation).
- **B:** Ship the full ICO questionnaire schema now. Needs
  legal sign-off on field shapes; would slip the sprint.

**Default at kickoff: Option A.** Same shape as S10's
Safety Case Summary decision — v1.0 best-effort; canonical
schema reconciliation → v1.1.

**All three Option A defaults proceed at kickoff per the
auto-mode discretion granted at the user's "continue"
instruction; pattern established cleanly through S10.**

## Top 3 risks (PAFM-SD Appendix E)

1. **UK GDPR statutory accuracy ambiguity at kickoff.**
   Same risk shape as S10 BSA 2022. SAR / DPIA / breach-log
   field names have legal weight; v1.0 ships best-effort
   doc-comment-anchored field shapes with `// GDPR ref:
   <article>` references for pre-pilot legal review.
   **Mitigation:** carry the S10 doc-comment pattern; mark
   field shapes "v1.0 best-effort" in DTO docs; pre-pilot
   legal review remains a pre-deploy step, not a v1.0
   blocker.
2. **Org-scoped vs project-scoped tenant filter inversion.**
   4 of 5 new entities (ROPA / SAR / breach log / retention
   schedule) are org-scoped (`OrganisationId ==
   _tenant.OrganisationId`); 1 is project-scoped (DPIA via
   project FK). The org-scoped shape is rarer in CIMS — only
   `Invitation` from S0 uses it currently. Risk: a wrong
   filter would leak data across tenants. **Mitigation:**
   model-level coverage test asserts each entity's filter is
   registered (existing `CimsDbContextTenantFilterTests`);
   T-S11-07 behavioural cross-tenant test exercises both
   filter shapes (project-scoped DPIA + at least one
   org-scoped entity).
3. **30-day SAR clock + 72-hour breach clock semantics.**
   Both clocks have legal implications if missed.
   **Mitigation:** v1.0 records timestamps and exposes
   computed DueAt fields; **automatic overdue flagging /
   notifications → v1.1 / B-NNN if pilot need surfaces.**
   Same shape as S10 / B-073 Gateway deadline tracking.
