## Sprint goal

Deliver the Risk & Opportunities module (PMBOK 5 Risk knowledge area)
on the foundation laid by S0 + S1. Covers the risk register with RBS
taxonomy, the standard 5×5 probability-impact matrix, qualitative and
quantitative assessment, Monte Carlo simulation for cost and schedule
contingency, contingency drawdown tracking, response plans, and the
mirror opportunity register.

Inherits from S0 + S1:
- `ITenantContext` and the full query-filter scoping. Every Risk and
  Opportunity entity is tenant-scoped via `Project.AppointingPartyId`.
- Two-tier role model (ADR-0010): risk endpoints are candidates for
  `[Authorize(Roles=...)]` global gates plus per-project
  `HasMinimumRole` checks.
- Audit-twin pattern (post-S1 PR #33). Every risk mutation produces a
  per-row Insert/Update/Delete via `AuditInterceptor` plus a
  structured `AuditService.WriteAsync` event with semantic detail —
  both committed atomically with the entity write.
- B-001 token revocation, B-002 rate limiting, ADR-0014 access-token
  SLA — all in force from the auth layer down.
- Transaction-wrap pattern (PRs #29 / #34 / #35) — any service method
  that does ExecuteUpdateAsync + a separate SaveChanges in one
  logical operation must `BeginTransactionAsync` round both.

## Top 3 risks (PAFM-SD Appendix E)

1. **Monte Carlo determinism for tests.** A naive `Random.Next()`
   makes every test run produce different P10/P80/P90 numbers — flaky
   CI. Mitigation: use `Random(seed)` with a fixed seed in tests,
   document the seed in the test fixture; production uses the
   parameterless constructor. Behavioural tests assert on percentile
   *bounds*, not exact values, where feasible. The 1000-iteration
   floor is enough for percentiles to stabilise within ~1%.
2. **3-point estimate semantics easy to get backwards.** Best/most
   likely/worst can be entered with best > worst by mistake (a
   negative spread), or distribution-shape rules (e.g. PERT mean
   formula) can be misordered. Mitigation: validate
   `BestCase ≤ MostLikely ≤ WorstCase` at write time; PMBOK fixture
   tests for each distribution shape against textbook expected values.
3. **Risk score interpretation.** A 5×5 matrix gives scores 1–25, but
   industry conventions vary on which scores trigger which response
   thresholds (e.g. some models use 1–4 low / 5–9 medium / 10–25 high;
   others 1–6 / 7–14 / 15–25). Mitigation: keep the score numeric
   only; do NOT bake a default threshold mapping into the entity.
   Threshold configuration belongs to S14 Admin Console (per-tenant
   setting). Document this explicitly in `RiskMatrix` doc-comments.
