## Sprint goal

Deliver formal procurement workflow per PAFM-SD F.7: project-level
procurement strategy capture, tender package creation, weighted
evaluation matrix (price + quality), award workflow with atomic
Contract spawn (re-using the S5 Approve→Variation pattern),
NEC4-aligned early warnings + compensation events. Full audit
trail via the existing audit-twin pattern.

S6 is the **second test for the S5-retro cadence model**: the
"workflow domain compresses when canonical" hypothesis. S5 showed
that NEC4-aligned change control compressed cleanly. S6 is more
domain-broad (5 entities + 1 evaluation matrix algorithm + 1 NEC4
state machine) but still standardly-formalised. **Predict:
same-session compression.** If S6 *doesn't* compress, that's the
first concrete cadence-breaker.

Inherits from S0..S5:
- Tenant query filter on every entity via Project.AppointingPartyId.
- Two-tier role model (ADR-0010): write paths gated `TaskTeamMember+`
  for raise/submit; `ProjectManager+` for award / contract / decision.
- Audit-twin pattern (PR #33).
- `Core/<X>.cs` pure-function pattern for the evaluation matrix
  scoring + the compensation-event state machine.
- Approve-with-spawn pattern from S5 (Approve→Variation) re-applied
  for Award→Contract atomic spawn.
- v1.1-candidate decisions land inline (S5-retro action) — every
  v1.1 deferral in this kickoff has a B-NNN entry written in the

## Top 3 risks (PAFM-SD Appendix E)

1. **NEC4 compensation-event state machine subtlety.** NEC4 clause
   60.1 lists 19 events that constitute compensation events; the
   workflow has formal time deadlines (PM 4 weeks; contractor 3
   weeks) and deemed-acceptance rules. v1.0 ships the 5-state
   forward path (Notified → Quoted → Accepted | Rejected →
   Implemented) without the deadline-driven automatic
   transitions. Mitigation: deadline-driven automation deferred
   to B-048 / B-049 / B-050; v1.0 ships honest manual workflow
   with audit trail. The 5-state machine itself is canonical.
2. **Evaluation matrix weight-sum invariant.** Weights per
   package must sum to 1.0 (or 100 if expressed as percentages).
   Floating-point summation across multiple criteria can drift.
   Mitigation: store weights as `decimal(5,4)` and validate
   that abs(sum - 1.0) < 0.0001 at write time. Test fixtures
   include the boundary cases.
3. **Award workflow side-effect coupling.** Awarding a Tender
   must (a) transition the winning Tender to Awarded, (b)
   transition all other Tenders on the same package to Rejected
   automatically, (c) close the TenderPackage, (d) spawn a
   Contract atomically. Mitigation: single transactional
   SaveChanges call with all four mutations + 3 audit events;
   reuse the S5 ApproveAsync transactional pattern.

## Cadence prediction (S5-retro action explicit test)

S5-retro predicted: "Approve→Contract pattern from S5 will likely
re-apply; workflow-shaped domain again — predict same-session
compression; if it doesn't compress, that's the cadence-breaker
we've been waiting for."

The honest break-points to watch:
- **NEC4 compensation events workflow design** (T-S6-08) — if I
  find myself re-reading clause 60.1 to design the state machine
  rather than copying the canonical 5 states, that's the genuine
  interpretive moment.
- **Evaluation matrix weighting semantics** — does weight 0.5
  mean "50% of the score comes from this criterion" or "this
  criterion's score is multiplied by 0.5 and added"? They're
  the same arithmetically when weights sum to 1.0; v1.0 enforces
  the sum-to-1.0 invariant so the question doesn't surface
  beyond docs.
