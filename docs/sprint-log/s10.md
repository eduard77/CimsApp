Deliver the F.10 Golden Thread / BSA 2022 module: HRB
(Higher-Risk Building) flagging, Gateway 1/2/3 submission
packages, Mandatory Occurrence Reporting (MOR), Safety Case

specifically for HRBs (residential buildings ≥ 18m or 7
storeys). v1.0 ships the data shapes + workflows for those
regimes; **statutory legal accuracy review is not in

+ legal sign-off step before pilot deploy.

## Decisions required at kickoff

**Decision 1 — Golden Thread immutability semantics.**

The BSA 2022 Golden Thread requirement is "the building's
information is preserved + accessible throughout its life,
with an audit trail of who changed what and when." Two
plausible v1.0 interpretations:

- **A — Soft immutability (recommended).** Add
  `IsInGoldenThread` (bool) to `Document`. Service-layer
  guards block edit / state-transition / soft-delete on
  GT-marked documents (except via a GT-specific "amend"
  flow that preserves history). Existing AuditInterceptor
  provides the per-row audit. v1.0-honest.
- **B — Hard immutability.** Append-only DB writes; no
  UPDATE / DELETE on GT-marked rows; new revisions are
  always-new rows. Massively more work; touches every
  service.
- **C — Defer to v1.1 / B-NNN.** Ship the GT *flag* in v1.0
  but no enforcement of immutability; flag is a tag for the
  audit-export step.

**Default at kickoff: Option A.** Provides genuine
immutability for the v1.0 pilot pre-customer-deploy, without
the rewrite-every-service cost of Option B. Option C is too
weak to claim "Golden Thread repository" in good faith.

**Decision 2 — Mandatory Occurrence Reporting BSR push.**

BSA 2022 requires HRB occurrences to be reported to the
Building Safety Regulator. The BSR currently has no public
API for programmatic push (as of 2026-05-02; verify at
implementation time).

- **A — v1.0 captures the report; manual push to BSR via
  email / regulator portal (recommended).** Aligns with
  current BSR practice; v1.1 / B-070 wires up the API when
  it exists.
- **B — Build a stub BSR adapter and document it as a
  v1.1 swap.** Adds complexity for no v1.0 value.

**Default at kickoff: Option A.**

**Decision 3 — Safety Case Summary scope.**

BSA 2022 Safety Case is a substantial document covering
the building's entire safety strategy. v1.0 cannot ship the
canonical Schedule 5 structure without legal review.

- **A — JSON aggregator best-effort inference (recommended).**
  Same shape as T-S7-03 MPR. Pulls together: HRB metadata,
  open Risks, Mandatory Occurrence Reports, Gateway package
  status, key documents in Golden Thread. v1.1 / B-071
  reconciles against PAFM-CIMS-001 PDF Schedule 5 wording.
- **B — Defer Safety Case Summary entirely to v1.1.** Trims
  the F.10 bullet by 1 — but the sprint feels incomplete
  with one bullet visibly missing.

**Default at kickoff: Option A.**

## Top 3 risks (PAFM-SD Appendix E)

1. **BSA 2022 statutory accuracy ambiguity at kickoff.**
   Several MOR / Gateway / Safety Case fields have legal
   meaning that v1.0 ships best-effort. Wrong field shapes
   can be patched later, but wrong field NAMES that
   customers rely on become harder to change without breaking
   integrations. **Mitigation:** anchor every MOR / Gateway
   / Safety Case field to a `// BSA 2022 ref: <section>`
   doc-comment so v1.1 / pilot legal review can map field-
   by-field. Mark fields as "v1.0 best-effort" in DTO docs.
2. **Golden Thread immutability scope creep.** Option A
   (soft immutability) is the kickoff default but the
   service-layer-guard surface is wide — every service that
   touches Document needs a "is this GT? if so, refuse"
   check, OR the check goes in the AuditInterceptor /
   DbContext. **Mitigation:** put the check in
   `DocumentsService` only (Edit / Transition / SoftDelete
   paths); other services that touch Document indirectly
   (e.g. RfiDocument FK adds) are NOT blocked in v1.0 — the
   immutability is on Document content, not on relationships.
3. **MidpEntry / TidpEntry sign-off interaction with GT.**
   S9 added MIDP / TIDP. A signed-off TIDP entry that
   references a GT-marked Document creates a cross-domain
   ordering question: which freezes first? **Mitigation:**
   v1.0 treats them as independent — TIDP sign-off freezes
   the TIDP entry; GT marking freezes the Document; no
   coupling. If real workflows ever need coupling
   (e.g. "GT-marking a Document requires its TIDP entry to
   be signed off"), B-NNN candidate.
