## Module DoD (predicted, pending .docx reconciliation)

- Global search across the high-traffic project-scoped entity
  types (documents, RFIs, actions, risks, change requests,
  early warnings, compensation events) — single endpoint
- Filter by entity type

## Decisions required at kickoff

Three triple-decision Option A defaults — pattern carried from
S10/S11/S12/S14.

### Decision 1 — Search backend
- **Option A (default, lean):** **EF Core `EF.Functions.Like` /
  `Contains` against existing tables.** Per-entity provider
  runs a project-scoped LIKE query within the tenant query
  filter; aggregator merges + ranks by weighted match. Zero
  new infra. Works identically on LocalDB (dev),
  `mssql/server:2022-latest` (CI), and any prod SQL Server.
  Upgrade path to SQL Server full-text catalogues → v1.1 /
  B-095 when pilot search latency becomes user-visible.
- Option B: SQL Server full-text catalogues. Faster, ranked,
  language-aware — but LocalDB doesn't ship FTS by default,
  so dev environment friction. Migration adds
  `CREATE FULLTEXT INDEX` per searchable table. ~6h
  implementation + dev-environment-setup doc.
- Option C: Separate Lucene / Elasticsearch process. Best
  search quality but biggest infra cost; explicit non-goal
  for v1.0 small-pilot scale.
- **Choice:** **Option A.** B-095 unblock condition: "first
  pilot project where end-to-end search latency exceeds 500ms
  on a project with > 1000 rows in any one searchable table".

### Decision 2 — Cross-entity vs per-entity endpoint
- **Option A (default, lean):** **Single cross-entity
  endpoint.** `GET /api/v1/projects/{projectId}/search?q=...&types=...`
  returns a unified hit list with `(EntityType, Id, Title,
  Snippet, Score)` discriminator. Pilot users want one search
  box, not seven. Per-type filtering via optional `types=`
  query param.
- Option B: One endpoint per entity type
  (`/projects/{id}/documents/search`, `.../rfis/search`,
  etc.). Simpler implementation but worse UX.
- **Choice:** **Option A.** v1.0 covers the seven entity types
  listed in the DoD; pilot-driven extension to additional
  types (stakeholders, MIDP, TIDP, lessons-learned,
  opportunities, inspections, alert rules) → v1.1 / B-097.

### Decision 3 — Saved searches / search history
- **Option A (default, lean):** **No persistence in v1.0.**
  Fresh search each session. Search history (browser-local
  recent queries) is a UI concern not a service-layer one;
  the v1.0 service ships stateless. Saved-searches feature
  (named, shareable, with notification on new hits) → v1.1 /
  B-096.
- Option B: Persist. Adds `SavedSearch` entity + service +
  UI. Pulls notification wiring through to surface
  matching-row alerts (overlap with S14 AlertRule shape).
- **Choice:** **Option A.** B-096 unblock condition: "pilot
  user explicitly requests saved-search functionality OR
  evidence that users are repeatedly running the same query
  manually".

## Top-3 risks (CATEGORY only — S9-retro pattern)

1. **SQL injection / LIKE wildcard escaping.** User-supplied
   `%`, `_`, `[` characters in the query string would be
   interpreted as LIKE wildcards if passed raw. Mitigation:
   escape via `EF.Functions.Like(column, "%" + Escape(q) + "%",
   '\\')` with explicit escape character; per-provider tests
   exercise each wildcard character to confirm literal match.
2. **Cross-tenant leak via search.** Per-entity providers must
   honour the existing query filter (project-scoped via
   `Project.AppointingPartyId`). Mitigation: every provider
   query goes through the standard `db.<DbSet>` (which carries
   the global filter); one tenant-isolation test per provider
   confirms a search from tenant A doesn't return tenant B's
   rows. Background-service-tenant pattern from S14 does NOT
   apply here (search is request-scoped — `ITenantContext` is
   live).
3. **Performance on large projects.** Naïve LIKE across seven
   entity types could O(N×7) at scale. Mitigation: take-N per
   provider (default 10) + early-exit aggregator + document
   the scale ceiling explicitly via B-095. Pilot-size project
   (~hundreds of rows per entity) is comfortable; B-095 is
   the real fix.
