## Sprint goal

Deliver the Schedule & Programme module — the first sprint with
**genuinely novel algorithmic complexity** since S0. CPM forward /
backward pass, total / free float, critical-path identification.
Baselines that snapshot a schedule and compare it to actuals.
Last Planner System boards (Lookahead, Weekly Work Plan, PPC).
Takt planning for repetitive work. MS Project XML
import / export (the v1.0 interop that makes this module usable
on real projects). Gantt and network-view data endpoints.

S4 is explicitly the test for the S3-retro prediction:
> "S4 Schedule & Programme has CPM algorithms that are genuinely
> novel relative to what the codebase has shipped." If S4 still
> closes same-day with a 1.5×-trimmed estimate, the
> "AI-paired-on-pattern-similar-work" cadence model is wrong.

Inherits from S0 + S1 + S2 + S3 (every prior pattern):
- `ITenantContext` and the full query-filter scoping. Every new
  entity is tenant-scoped via `Project.AppointingPartyId`.
- Two-tier role model (ADR-0010): write paths gated
  `TaskTeamMember+` for floor-level entry, `ProjectManager+` for
  baseline snapshots and other sensitive transitions.
- Audit-twin pattern (PR #33).
- `Core/<X>` pure-function pattern from `Core/Evm.cs` /
  `Core/RiskMatrix.cs` / `Core/StakeholderMatrix.cs` —
  `Core/Cpm.cs` will be the new addition, fully testable without
  DB.
- Three-commit-per-task cadence (entity-with-service / migration
  / tests).
- All hardening from the post-S0..S3 sweeps (token revocation,
  rate limiting, body-binding fixes, audit-twin atomicity,
  transaction-wrap pattern, NoAction FKs everywhere).

## CR-005 — proposed deferrals

Three options for the user, in order of recommendation:

**Option A — Defer MS Project XML export, Takt, network view (~17h × 1.5 = ~26h).**
Trimmed estimate: ~68h. Within capacity.
- T-S4-10 MS Project XML export → `B-031` v1.1 (import alone is
  more valuable for "get my existing plan into CIMS" use case;
  export round-trips back to MS Project, which most v1.0 users
  won't need on day 1).
- T-S4-08 Takt planning → `B-032` v1.1 (Takt is repetitive-work
  optimisation — high-rise, infrastructure schemes; a v1.0

  it).
- T-S4-11 network-view endpoint → keep Gantt only; defer
  network-view to `B-033` v1.1 (Gantt is the dominant
  visualisation in the UK industry; network diagrams are a
  PMBOK-textbook view that real PMs rarely use).

**Option B — Defer Takt and LPS Weekly Work Plan / PPC (keep Lookahead) (~10h × 1.5 = ~15h).**
Trimmed estimate: ~80h. 8h over capacity but inside the S2 /
S3 same-day-velocity envelope.
- T-S4-08 Takt → `B-032` v1.1.
- T-S4-07 partial: keep `LookaheadEntry`, defer `WeeklyWorkPlan`
  + `WeeklyTaskCommitment` + PPC to `B-034` v1.1. Rationale:
  Lookahead is the LPS feature most v1.0 users will set up
  first; Weekly Work Plan + PPC are the ongoing-management
  features that come into use at week 4+ of a project.

**Option C — No deferral; absorb the 22.5h overrun.**
Honest framing: the post-S2-retro reality is that the formal
capacity calculation does not predict actual effort. If S4
retains the S0..S3 cadence, 94.5h fits in one session. If it
**doesn't**, the retro will document it as the predicted
break-point.

**My recommendation: Option A.** S4 ships the core scheduler
(CPM + baselines + LPS + Gantt + MS Project import) in v1.0,
which is a complete-feature in itself. The three deferred items
(Takt, MS Project export, network view) all have natural v1.1
homes and don't block any other v1.0 module.

## Top 3 risks (PAFM-SD Appendix E)

1. **CPM correctness.** Forward / backward pass, lag handling,
   constraint propagation, cycle detection — every textbook
   CPM has subtle edge cases (zero-duration milestones, FF lag
   on parallel paths, calendar-day vs working-day duration
   semantics). Mitigation: textbook PMBOK fixture-based tests
   in `Core/Cpm.Tests.cs`; explicit "Day" duration unit only in
   v1.0 (no calendar / non-working-day complications); cycles
   rejected at the dependency-add layer rather than swallowed.
2. **MS Project XML interop fragility.** Microsoft Project's
   XML format is not stable across MSP versions; importer needs
   to be lenient on optional elements but strict on the core
   `<Tasks>` / `<PredecessorLink>` shape. Mitigation: import is
   into-empty-only (no merge); fixture files from MSP 2019 +
   MSP 365 cover the realistic v1.0 input range; export emits
   the well-known core elements, not the every-MSP-feature
   superset.
3. **The "S4 may not match the S0..S3 same-day cadence"
   prediction.** If CPM lands in 12h estimated and actually
   takes 4-6h (the S3 cadence), the prediction is wrong. If it
   takes 12-20h, the prediction is right and the model needs
   updating. Mitigation: time-box T-S4-04 explicitly to 1
   working-day-equivalent; if not done by then, **stop and
   write a retrospective day-end entry before moving on**.
