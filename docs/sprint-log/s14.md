## Decisions required at kickoff

Three triple-decision Option A defaults — pattern carried from S10/S11.

### Decision 1 — SMS transport (Twilio) in v1.0?
- **Option A (default, lean):** Defer SMS to v1.1 / B-090. The .docx
  explicitly marks SMS as "optional"; SignalR + SMTP cover the v1.0
  notification surface. Twilio adds a paid third-party dependency,
  account provisioning, and a non-trivial test shape (mocking the
  Twilio SDK or webhook flow). Pilot users in UK construction settings
  reach in-app + email reliably; SMS is fast-follower territory.
- Option B: Ship SMS now. Adds Twilio NuGet package (Ch 24.6 24-hour
  wait) + Twilio account spin-up + ~6h of integration test surface.
- **Choice:** **Option A.** B-090 carries explicit unblock condition
  "Twilio account provisioned + SMS delivery test in staging".

### Decision 2 — Email queue persistence
- **Option A (default, lean):** In-memory background queue via
  `IHostedService` + `Channel<T>`. Simple, zero-package, fits within
  the existing service container shape. Email loss on app restart is
  acceptable for v1.0 (notifications are advisory, not state-bearing
  — the underlying record in cost/schedule/risk is the source of
  truth). Persistent queue → v1.1 / B-091.
- Option B: Persist outbound email to a `OutboundEmail` table with a
  background worker that drains it. Survives restarts, supports
  retry-with-backoff, but doubles the entity surface and adds a
  delivery-state machine.
- **Choice:** **Option A.** B-091 unblock condition "first incident
  where missed restart-time email caused user-visible regression OR
  pilot user requests delivery audit trail".

### Decision 3 — AlertRule shape
- **Option A (default, lean):** Single `AlertRule` entity with
  `(Metric enum, Comparison enum, Threshold decimal, Recipients
  GlobalRole|UserId)`. Three metrics in v1.0: `CostVariance`,
  `ScheduleSlip`, `RiskEscalation`. Background evaluator runs every
  N minutes, queries the relevant aggregate per project, fires
  `Notification` + `IEmailSender.SendAsync` when threshold crossed.
  No re-firing within cooldown window. Rich rule DSL → v1.1 / B-092.
- Option B: Expression-based rule DSL (boolean combinations,
  per-project custom rules, time-of-day windows). Adds a parser/
  evaluator + UI editor.
- **Choice:** **Option A.** B-092 unblock condition "pilot user
  requests rule combinations that cannot be expressed as a single
  metric/comparison/threshold tuple".

## Top-3 risks (predict CATEGORY only — S9-retro pattern)

1. **Background-service lifecycle / DI scope.** `IHostedService` +
   scoped `CimsDbContext` + tenant context interaction. Mitigation:
   create scope per-iteration in evaluator; do NOT inject scoped
   services directly into the singleton hosted service.
2. **SignalR + JWT auth wiring.** The bearer token in the
   `access_token` query string vs `Authorization` header is a
   well-known SignalR rough edge. Mitigation: configure
   `JwtBearerOptions.Events.OnMessageReceived` to read the
   `access_token` query param when the path starts with the hub
   route.
3. **AlertRule evaluator interaction with multi-tenancy.** The
   evaluator iterates across tenants, so it must explicitly NOT use
   the request-scoped `ITenantContext`. Mitigation: enumerate
   organisations explicitly; evaluate per-org with
   `IgnoreQueryFilters()` + manual filter, or set tenant context per
   iteration via a dedicated `ISystemTenantContext`.
