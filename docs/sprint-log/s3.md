## Sprint goal

Deliver the Stakeholder & Communications module (PMBOK 7
Stakeholders performance domain). Covers the stakeholder register
with power/interest scoring, per-stakeholder engagement plan,
engagement log of interactions, and the project-level
communications matrix.

Inherits from S0 + S1 + S2:
- `ITenantContext` and the full query-filter scoping. Every new
  entity is tenant-scoped via `Project.AppointingPartyId`.
- Two-tier role model (ADR-0010): write paths gated `TaskTeamMember+`
  with `ProjectManager+` for sensitive transitions.
- Audit-twin pattern (PR #33) — every mutation produces a per-row
  Insert/Update/Delete via `AuditInterceptor` plus a structured
  `AuditService.WriteAsync` event committed atomically.
- Score-via-P×I matrix pattern from S2 — apply to the
  Power/Interest grid (Mendelow) the same way the 5×5 risk matrix
  shipped in T-S2-05.
- All hardening from the post-S1 + post-S2 sweeps stays in force
  (token revocation, rate limiting, body-binding fixes,
  audit-twin atomicity, transaction wraps).

## Top 3 risks (PAFM-SD Appendix E)

1. **Power/Interest scale convention varies across PMBOK editions
   and industry references.** PMBOK 6/7 use 1-5; some references
   use 1-10; some use qualitative low/medium/high. Mitigation:
   pick 1-5 (consistent with S2's risk matrix), document the
   choice in `Core/StakeholderMatrix.cs` doc-comments. Threshold
   mapping (Mendelow quadrants — ManageClosely / KeepSatisfied /
   KeepInformed / Monitor) is computed at fixed 3-as-the-midpoint:
   Power ≥ 3 + Interest ≥ 3 → ManageClosely, etc. Document as a
   per-tenant override candidate for S14 Admin Console.
2. **Engagement log throughput.** Long-running projects produce
   hundreds of log entries; the listing endpoint must paginate or
   bound. Mitigation: v1.0 hard-bounds the list to the most
   recent 200 entries per stakeholder; older entries accessible
   via dated filter. v1.1 candidate: full pagination if needed.
3. **Communications matrix is quasi-templated** — every project
   repeats ~80% of the same items (MPR, daily diary, variation
   notices, etc.). Mitigation: per-project storage in v1.0,
   accept duplication; v1.1 candidate: org-level templates
   that seed new projects, similar to the existing project
   provisioning mechanism for documents.
