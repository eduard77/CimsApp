ARCHITECTURAL REFACTOR (the headline change):

  PR #33 refactor(audit) — make audit-twin events atomic
    AuditService.WriteAsync no longer calls SaveChangesAsync;
    it just adds the AuditLog row to the change tracker. 28
    call sites flipped so audit.WriteAsync runs BEFORE
    db.SaveChangesAsync, and a single SaveChanges commits
    both the entity write AND the structured audit-twin
    event in one transaction. Before the refactor, every
    business mutation produced TWO transactions (entity +
    audit) — a process crash between them could leave the
    entity persisted but the structured event missing,
    breaking the audit-twin contract. Forensic adequacy was
    not at risk (per-row AuditInterceptor was always atomic
    with the entity write); discoverability is now fully
    closed too. Side effect: one fewer SaveChanges
    round-trip per mutation.

THREE SAME-SHAPE TRANSACTION-WRAP BUGS (one pattern, three
sites — found by following the trail from each fix to the
next):

  PR #29 fix(register) — RegisterAsync wraps User.Add +
    InvitationService.MarkConsumedAsync (ExecuteUpdateAsync)
    in BeginTransactionAsync. Closes the
    "one-invitation-two-users" crash window where a process
    crash between the two writes would leave the User
    visible but the Invitation still consumable.

  PR #34 fix(document-transition) — DocumentsService.TransitionAsync
    wraps the revision-publish ExecuteUpdateAsync (sets
    PublishedAt / ApprovedById / Suitability on latest
    revisions) and the doc.CurrentState SaveChanges in one
    transaction. Pre-fix: revisions could be marked Published
    while the Document.CurrentState lagged behind.

  PR #35 fix(org-create) — OrganisationsController.Create
    wraps the org save and the bootstrap-invitation save
    (via InvitationService.CreateAsync) in one transaction.
    Pre-fix: a crash between the two would leave the org Code
    reserved by the unique index but no bootstrap invitation,
    so the org was orphaned forever (retry impossible).

The pattern: any "ExecuteUpdateAsync followed by separate
SaveChanges in one logical operation" or "explicit save +
service call that also saves" is two SQL transactions. None
of these have a live exploit path during the v1.0 internal
pilot — the crash window is sub-millisecond and the
single-developer flow doesn't race — but the wraps are
correct for any deployment with concurrent traffic or
infrastructure flakiness. EF in-memory provider treats
BeginTransactionAsync as a no-op (suppress the warning via
ConfigureWarnings(w => w.Ignore(InMemoryEventId.TransactionIgnoredWarning))
if a test exercises a tx-wrapped path).

A future audit pass should grep for `ExecuteUpdateAsync` and
verify each site is either (a) the only write in its method
or (b) wrapped with the surrounding writes in a transaction.
Both current sites (Invitations.MarkConsumedAsync and
DocumentRevisions inside TransitionAsync) are now covered.

ADDITIONAL PR LANDED 2026-04-29 LATE
-------------------------------------
PR #41 fix(audit) — AuditLog.UserId nullable for
anonymous-actor flows. Discovered when the smoke test
above (forward path #3) hit POST /api/v1/organisations
against the real DB and got HTTP 500. The bootstrap-
invitation audit row wrote Guid.Empty to UserId as a "no
actor" sentinel; SQL Server rejected the INSERT with FK
violation FK_AuditLogs_Users_UserId. Broken in production
for ~a week (since T-S0-11 / PR #11) but 100% latent
under unit tests because EF in-memory ignores FK
constraints.

Fix: AuditLog.UserId becomes Guid? + User? navigation;
AuditService.WriteAsync takes Guid? userId; three call
sites that did `?? Guid.Empty` pass the nullable through.
Query filter widened to `x.User != null && x.User.OrganisationId == _tenant.OrganisationId`
so anonymous-actor rows are hidden from tenant queries.
New migration AuditLogUserIdNullable.
