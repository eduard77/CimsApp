# Sprint S5 — Change Control

**Planned:** 2026-05-01 → 2026-05-22 (3 weeks)
**Roadmap reference:** PAFM-SD Ch 6.4, Phase 2
**Module DoD:** PAFM-SD Appendix F.6

## Sprint goal

Deliver formal change-control workflow per PAFM-SD F.6:
five-state ChangeRequest workflow (Raised → Assessed → Approved
| Rejected → Implemented → Closed), category tagging
(Scope/Time/Cost/Quality), BSA HRB categorisation tagging
(Cat A / B / C where applicable), role-gated approval, structural
links to S1 Variations + S4 programme impact, full audit trail
via the existing audit-twin pattern.

S5 is **explicitly the test for the S4-retro prediction**:
> "S5 will not close same-day, because the workflow shape is
> interpretive (what states / transitions / approval rules belong
> in v1.0?). If S5 still closes same-day, that's another data
> point to refine the cadence model."

If the spec drives a clean answer (well-formalised domain like
PMBOK CPM was in S4), the prediction falls and "novel decision"
isn't the cadence-breaker either. If the spec leaves real
ambiguity, the prediction holds.

Inherits from S0..S4:
- `ITenantContext` and the full query-filter scoping. Every new
  entity is tenant-scoped via `Project.AppointingPartyId`.
- Two-tier role model (ADR-0010): write paths gated
  `TaskTeamMember+` for raise; `ProjectManager+` or higher for
  approve / reject / implement / close transitions.
- Audit-twin pattern (PR #33) — every state transition produces
  a structured event committed atomically with the row mutation.
- `Core/<X>.cs` pure-function pattern from `Core/Cpm.cs` /
  `Core/MsProjectXml.cs` — `Core/ChangeWorkflow.cs` for the
  state-machine transition rules.
- Three-commit-per-task cadence (entity-with-service / migration
  / tests).

## Top 3 risks (PAFM-SD Appendix E)

1. **State-machine ambiguity around the Reject / Implement /
   Close path.** PMBOK is consistent on raise / assess / approve;
   "implement" and "close" semantics vary by source. v1.0
   interpretation (documented in `Core/ChangeWorkflow.cs`
   doc-comments): Approved → Implemented when the change has
   been actioned in delivery; Implemented → Closed when the
   resulting Variation / programme update / cost adjustment
   has been recorded. Reject is allowed from Raised or Assessed
   only — once Approved, only Implemented / Closed forward.
2. **BSA categorisation legal-text drift.** The Building Safety
   Act 2022 and subsequent secondary legislation use specific
   category labels (Cat A / B / C); the exact thresholds change
   as Building Safety Regulator guidance evolves. v1.0 ships
   the categorisation as an enum *tag* with no automatic
   inference — the PM marks the CR; reporting downstream uses
   the tag. v1.1 candidate: per-tenant configurable categorisation
   rules + automatic inference from change scope.
3. **The S4-retro prediction.** S4 retro predicted S5 won't
   match the same-day cadence because the workflow shape is
   "interpretive". Mitigation: write the workflow against
   PMBOK / BSA-aligned states without litigating each
   transition; if a real interpretive decision surfaces
   mid-sprint, document it inline in
   `Core/ChangeWorkflow.cs` rather than expanding scope.
